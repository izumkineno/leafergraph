这份文档面向 `packages/leafergraph` 的维护者，目标是把当前主包的真实装配链、目录职责和阅读顺序讲清楚。

## 1. 总体分层

当前主包的职责边界已经固定为：

| 子系统 | 主要职责 |
| --- | --- |
| `public` | 对外实例入口与 facade 聚合 |
| `api` | 公开 API 的内部 controller 与 host helper |
| `graph` | 图装配、恢复、刷新、主题、本地运行反馈投影 |
| `interaction` | 拖拽、缩放、折叠、连接预览、交互提交 |
| `node` | 节点壳、布局、端口、节点运行时 façade |
| `link` | 正式连线路径和数据流动画 overlay |
| `widgets` | 主包内部复用的 Widget foundation 图元 |

同时还要记住这些外部真源边界：

- `@leafergraph/core/node`
  - 图模型、节点定义、模块、注册表
- `@leafergraph/core/execution`
  - 执行内核、图级执行状态机、执行反馈、内建执行节点
- `@leafergraph/core/contracts`
  - 公共宿主协议、图 API 输入输出、Widget 契约
- `@leafergraph/core/widget-runtime`
  - Widget 注册表、编辑宿主、renderer lifecycle

一个实用判断是：

- “这是不是公开实例 API 的一部分？”先看 `public` 和 `api`
- “这是不是图装配、恢复、刷新或反馈投影？”先看 `graph`
- “这是不是节点壳、连线、Widget 的具体场景表达？”再看 `node / link / widgets`
- “这是不是手势、命中、选择和连接预览？”看 `interaction`

## 2. `src/graph` 目录树

`src/graph/` 已经从旧的平铺文件整理成固定目录树：

```text
src/graph/
  assembly/
    entry.ts
    runtime.ts
    scene.ts
    widget_environment.ts
  feedback/
    local_runtime_adapter.ts
    projection.ts
  host/
    bootstrap.ts
    canvas.ts
    mutation.ts
    restore.ts
    scene.ts
    scene_runtime.ts
    view.ts
  theme/
    host.ts
    runtime.ts
  history.ts
  style.ts
  types.ts
```

目录职责固定为：

- `assembly/`
  - 只负责装配顺序和对象接线
- `feedback/`
  - 只负责运行反馈适配与投影
- `host/`
  - 只负责图场景宿主、恢复、变更、视口与画布
- `theme/`
  - 只负责主题状态和主题切换后的协调刷新
- 根级 `history.ts / style.ts / types.ts`
  - 只保留图子系统共享的纯类型、样式和 history helper

补充约束：

- `assembly/widget_environment.ts` 是 Widget 基础环境的装配工厂，不再使用旧的 `*host` 命名
- 旧的 execution shim 已删除；图级执行真源直接来自 `@leafergraph/core/execution`
- 主包内部统一使用 `graphExecutionHost` 字段名，不再保留旧 execution 字段名

## 3. 运行时总装配链

主包从 `src/public/leafer_graph.ts` 进入，真实装配链如下：

```text
src/public/leafer_graph.ts
  -> src/graph/assembly/entry.ts
    -> src/graph/assembly/runtime.ts
      -> src/graph/host/canvas.ts
      -> src/graph/assembly/widget_environment.ts
      -> NodeRegistry
      -> src/graph/assembly/scene.ts
        -> src/graph/host/scene_runtime.ts
        -> src/node/runtime/controller.ts
        -> src/interaction/graph_interaction_runtime_host.ts
        -> @leafergraph/core/execution/LeaferGraphGraphExecutionHost
        -> src/graph/host/restore.ts
      -> src/graph/host/bootstrap.ts
      -> src/graph/feedback/local_runtime_adapter.ts
      -> src/api/graph_api_host.ts
```

这条链里需要特别记住两个点：

- `@leafergraph/core/execution` 是图级执行状态机和执行反馈的真源
- 主包只负责把 execution 状态投影回场景，并继续向外暴露 `RuntimeFeedbackEvent`

### 3.1 `assembly/entry.ts`

职责：

- 创建默认 `graphState`
- 准备默认样式、主题和布局参数
- 调用 `createLeaferGraphRuntimeAssembly(...)`
- 建立 `ready` Promise

这是入口级默认配置层。构造函数默认值和初始 runtime 壳面优先看这里。

### 3.2 `assembly/runtime.ts`

职责：

- 创建画布宿主
- 创建 Widget 基础环境
- 创建 `NodeRegistry`
- 创建场景运行时装配结果
- 创建 bootstrap host
- 创建 local runtime adapter
- 创建最终 `LeaferGraphApiHost`

它是“总装配器”，但不直接做节点、连线或交互细节。

### 3.3 `assembly/widget_environment.ts`

职责：

- 消费 `@leafergraph/core/widget-runtime`
- 创建 `LeaferGraphWidgetRegistry`
- 创建 `LeaferGraphThemeHost`
- 创建 `LeaferGraphWidgetEditingManager`

这层只做主包侧的 Widget 基础环境装配，不直接挂节点，也不直接做场景刷新。

### 3.4 `assembly/scene.ts`

职责：

- 串联节点、连线、Widget、交互、主题、恢复和执行宿主
- 让外层只看到一个已经接线完成的场景运行时集合

这里也是 `graphExecutionHost` 被创建并回传到总装配器的地方。

### 3.5 `host/bootstrap.ts`

职责：

- 安装 `modules`
- 安装 `plugins`
- 最后恢复 `document`

固定顺序是：

1. 模块
2. 插件
3. 文档恢复

### 3.6 `src/api/graph_api_host.ts`

职责：

- 把内部运行时宿主收口成稳定 facade
- 暴露 `use / installModule / registerNode / registerWidget`
- 暴露 `replaceGraphDocument / applyGraphOperation / applyGraphDocumentDiff`
- 暴露 `play / step / stop / playFromNode`
- 暴露 `subscribe*` 和查询 API

这层不再承担内部装配判断或场景细节拼装。

## 4. 各子系统职责地图

### 4.1 `graph`

`graph` 子系统内部再细分为四组职责：

| 目录或文件 | 主要职责 |
| --- | --- |
| `assembly/runtime.ts` | 总装配 |
| `assembly/scene.ts` | 场景级宿主接线 |
| `assembly/widget_environment.ts` | Widget 基础环境装配工厂 |
| `feedback/local_runtime_adapter.ts` | 执行反馈与节点状态反馈合流 |
| `feedback/projection.ts` | `RuntimeFeedbackEvent` 到主包宿主投影 |
| `host/scene_runtime.ts` | 场景刷新 facade，统一局部刷新和正式图变更 |
| `host/mutation.ts` | 节点/连线正式增删改移 |
| `host/restore.ts` | 整图恢复 |
| `host/view.ts` | 视口、坐标、选中态、`fitView()` |
| `theme/host.ts` | 主题模式状态 |
| `theme/runtime.ts` | 主题切换后的场景刷新协调 |

判断规则：

- 会不会改正式图状态或驱动场景刷新，优先看 `graph`

## 8. 当前维护约束

做主包改动时，始终检查这几个问题：

### 8.1 这个改动是不是放错层了

先问：

1. 这是模型定义吗
2. 这是执行内核吗
3. 这是 Leafer host / graph facade 吗
4. 这是 editor 壳层吗

如果是第 1 种，回 `@leafergraph/core/node`。  
如果是第 2 种，回 `@leafergraph/core/execution`。  
如果是第 4 种，不要塞回主包。

### 8.2 这个宿主是不是承担太多职责了

如果一个文件开始同时做：

- 场景装配
- 正式图变更
- DOM 交互
- 外部协议适配

那通常就该继续拆层。

### 8.3 这个问题是刷新链还是执行链

主包里这两条链经常被混淆：

- 刷新链
  - 节点壳、连线、Widget、渲染请求
- 执行链
  - `@leafergraph/core/execution` 状态机、节点执行、传播、反馈

排查前先分清是哪一条，能少走很多弯路。
