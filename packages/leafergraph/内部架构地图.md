# 内部架构地图

这份文档面向 `packages/leafergraph` 的维护者。

目标是把当前主包的真实装配链、目录职责和阅读顺序讲清楚，方便你：

- 新增能力时知道该放哪一层
- 排查 bug 时知道从哪条调用链切入
- 做重构时知道哪些边界不能轻易打穿

如果你还没读过包级 README，建议先回去看：

- [README.md](./README.md)

如果你在排查刷新或执行期行为，继续深链：

- [渲染刷新策略](./渲染刷新策略.md)

## 1. 总体分层

当前主包按六个源码子系统组织，但执行真源已经迁到 `@leafergraph/execution`，公共契约真源已经迁到 `@leafergraph/contracts`，Widget runtime 真源已经迁到 `@leafergraph/widget-runtime`，默认内容已经迁到 `@leafergraph/basic-kit`：

| 子系统 | 主要职责 |
| --- | --- |
| `api` | 运行时 facade、兼容转发层、对 contracts 的消费 |
| `graph` | 运行时装配、图状态、恢复、主题、视图、execution façade 装配 |
| `interaction` | 拖拽、缩放、折叠、连接预览、交互提交 |
| `node` | 节点壳、布局、端口、节点运行时 |
| `link` | 正式连线路径和数据流动画 overlay |
| `widgets` | Widget foundation 图元与接线；基础 widgets 已迁到 `@leafergraph/basic-kit` |

一个很实用的判断是：

- “这个能力是不是主包对外 API 的一部分？”先看 `api`
- “这个能力是不是图运行时装配或图状态宿主？”先看 `graph`
- “这个能力是不是节点、连线、Widget 的具体场景表达？”再看 `node / link / widgets`
- “这个能力是不是 DOM 指针交互和交互状态？”看 `interaction`

## 2. 运行时总装配链

主包从 `src/index.ts` 进入，但真正的内部装配链是：

```text
graph_entry_runtime
  -> graph_runtime_assembly
    -> graph_widget_runtime_host
    -> NodeRegistry
    -> graph_scene_runtime_assembly
      -> graph_scene_runtime_host
      -> node_runtime_host (host facade)
      -> interaction_host / interaction_runtime_host
      -> @leafergraph/execution
        -> node_execution_host
        -> graph_execution_host
      -> restore_host
    -> graph_bootstrap_host
    -> graph_api_host
```

### 2.1 `graph_entry_runtime.ts`

文件：

- `src/graph/graph_entry_runtime.ts`

职责：

- 创建默认 `graphState`
- 准备默认样式、主题和布局参数
- 调用 `createLeaferGraphRuntimeAssembly(...)`
- 建立 `ready` Promise

这是入口级默认配置层。  
如果你在想“构造函数默认值放哪”，先看这里。

### 2.2 `graph_runtime_assembly.ts`

文件：

- `src/graph/graph_runtime_assembly.ts`

职责：

- 创建画布宿主
- 创建 Widget 基础环境
- 创建 `NodeRegistry`
- 创建场景运行时装配结果
- 创建 bootstrap host
- 创建 local runtime adapter
- 创建最终 `LeaferGraphApiHost`

它是“总装配器”，但不直接做节点、连线或交互细节。

### 2.3 `graph_widget_runtime_host.ts`

文件：

- `src/graph/graph_widget_runtime_host.ts`

职责：

- 消费 `@leafergraph/widget-runtime`
- 创建 `LeaferGraphWidgetRegistry`
- 创建 `LeaferGraphThemeHost`
- 创建 `LeaferGraphWidgetEditingManager`

这层只做主包侧的 Widget 环境装配薄层，不直接挂节点、不直接做场景刷新。

### 2.4 `graph_scene_runtime_assembly.ts`

文件：

- `src/graph/graph_scene_runtime_assembly.ts`

职责：

- 接线节点、连线、Widget、交互、执行、恢复这几条真正依赖场景对象的宿主
- 让外层只看到一个已经串好的场景运行时集合

这是“场景级装配器”。  
如果你在想“某个宿主该不该感知 nodeViews/linkViews/canvasState”，通常就是这层负责接。

### 2.5 `graph_bootstrap_host.ts`

文件：

- `src/graph/graph_bootstrap_host.ts`

职责：

- 安装 `modules`
- 安装 `plugins`
- 最后恢复 `document`

固定顺序是：

1. 模块
2. 插件
3. 文档恢复

如果你要排查“为什么某个节点类型恢复时找不到”，优先检查这条顺序。

### 2.6 `graph_api_host.ts`

文件：

- `src/api/graph_api_host.ts`

职责：

- 把内部运行时宿主收口成稳定 facade
- 暴露 `use/installModule/registerNode/registerWidget`
- 暴露 `replaceGraphDocument/applyGraphOperation/applyGraphDocumentDiff`
- 暴露 `play/step/stop/playFromNode`
- 暴露 `subscribe*` 和若干交互/查询 API

这层不应该再承担“内部装配判断”或“场景细节拼装”。

## 3. 各子系统职责地图

## 3.1 `api`

先记住一个当前维护约束：

- `@leafergraph/contracts` 才是插件协议、图操作类型和图文档 diff 的正式真源
- `src/api/plugin.ts`、`src/api/graph_api_types.ts`、`src/api/graph_document_diff.ts` 现在只保留兼容转发职责
- `api` 目录在主包内部更应该被理解为“运行时 facade 层”，而不是公共类型真源

关键文件：

- `src/api/plugin.ts`
- `src/api/graph_api_types.ts`
- `src/api/graph_document_diff.ts`
- `src/api/graph_api_host.ts`

职责拆分：

| 文件 | 主要职责 |
| --- | --- |
| `plugin.ts` | 兼容转发到 `@leafergraph/contracts` 的插件与 Widget 契约 |
| `graph_api_types.ts` | 兼容转发到 `@leafergraph/contracts` 的图操作与反馈类型 |
| `graph_document_diff.ts` | 兼容转发到 `@leafergraph/contracts` 的纯文档 diff 工具 |
| `graph_api_host.ts` | 对外 facade |

判断规则：

- 对外可见的稳定类型，优先放 `@leafergraph/contracts`
- 内部运行时状态或宿主依赖，不放 `api`

## 3.2 `graph`

关键文件：

- `src/graph/graph_runtime_assembly.ts`
- `src/graph/graph_scene_runtime_assembly.ts`
- `src/graph/graph_scene_runtime_host.ts`
- `src/graph/graph_mutation_host.ts`
- `src/graph/graph_restore_host.ts`
- `src/graph/graph_execution_runtime_host.ts`
- `src/graph/graph_view_host.ts`
- `src/graph/graph_theme_host.ts`
- `src/graph/graph_theme_runtime_host.ts`

职责拆分：

| 文件 | 主要职责 |
| --- | --- |
| `graph_runtime_assembly.ts` | 总装配 |
| `graph_scene_runtime_assembly.ts` | 场景级宿主接线 |
| `graph_scene_runtime_host.ts` | 场景刷新 facade，统一局部刷新和正式图变更 |
| `graph_mutation_host.ts` | 节点/连线正式增删改移 |
| `graph_restore_host.ts` | 整图恢复 |
| `graph_execution_runtime_host.ts` | 图级 `play / step / stop` 状态机和 timer 管理 |
| `graph_view_host.ts` | 视口、坐标、选中态、`fitView()` |
| `graph_theme_host.ts` | 主题模式状态 |
| `graph_theme_runtime_host.ts` | 主题切换后的场景刷新协调 |

判断规则：

- “会不会改正式图状态或驱动场景刷新？”优先看 `graph`

## 3.3 `node`

关键文件：

- `src/node/node_host.ts`
- `src/node/node_shell_host.ts`
- `src/node/node_shell.ts`
- `src/node/node_layout.ts`
- `src/node/node_port.ts`
- `src/node/node_runtime_host.ts`

职责拆分：

| 文件 | 主要职责 |
| --- | --- |
| `node_host.ts` | 节点视图创建、刷新、销毁和与交互层的绑定入口 |
| `node_shell_host.ts` | 节点壳构建、缺失态和 resize 约束解析 |
| `node_shell.ts` | 节点壳图元结构 |
| `node_layout.ts` | 节点壳、端口区、Widget 区布局 |
| `node_port.ts` | 端口几何、锚点和布局计算 |
| `node_runtime_host.ts` | 节点快照、执行状态、输入输出值、Widget 动作回抛 |

判断规则：

- “这个问题是节点长什么样、怎么布局、怎么表达执行态？”先看 `node`

## 3.4 `link`

关键文件：

- `src/link/link.ts`
- `src/link/link_host.ts`
- `src/link/link_data_flow_animation_host.ts`

职责拆分：

| 文件 | 主要职责 |
| --- | --- |
| `link.ts` | 正式连线路径几何计算 |
| `link_host.ts` | 正式连线视图创建、移除和局部刷新 |
| `link_data_flow_animation_host.ts` | 数据流动画 overlay，不改正式连线路径 |

判断规则：

- “问题在正式连线本体”看 `link_host.ts`
- “问题在执行传播动画”看 `link_data_flow_animation_host.ts`

## 3.5 `widgets`

当前维护约束先记住：

- `@leafergraph/widget-runtime` 才是 Widget 注册表、lifecycle helper、编辑宿主、交互 helper 和默认 Widget 主题的正式真源
- `@leafergraph/basic-kit` 才是基础 widgets 的默认内容包

关键文件：

- `src/graph/graph_widget_runtime_host.ts`
- `src/widgets/foundation/*`

职责拆分：

| 文件 | 主要职责 |
| --- | --- |
| `@leafergraph/widget-runtime` | Widget 条目注册、renderer 调度、编辑宿主与交互 helper 真源 |
| `@leafergraph/basic-kit/widget` | 基础 Widget 内容库 |
| `widgets/foundation/*` | 主包内部复用的基础图元和样式支撑 |

判断规则：

- “Widget 值变了但视图没更新”先看 `@leafergraph/widget-runtime` 的 `widget_host.ts`
- “Widget 自定义 renderer 怎么接”先看 `@leafergraph/widget-runtime` 的 `widget_registry.ts + widget_lifecycle.ts`
- “文本编辑、菜单、焦点异常”先看 `@leafergraph/widget-runtime` 的 `widget_editing.ts`

## 3.6 `interaction`

关键文件：

- `src/interaction/graph_interaction_runtime_host.ts`
- `src/interaction/interaction_host.ts`
- `src/interaction/interaction_commit_source.ts`

职责拆分：

| 文件 | 主要职责 |
| --- | --- |
| `graph_interaction_runtime_host.ts` | 交互运行时壳面，收敛节点拖拽、resize、折叠、连接预览 |
| `interaction_host.ts` | DOM 指针生命周期和真实交互绑定 |
| `interaction_commit_source.ts` | 本地交互提交事件发射 |

判断规则：

- “有没有命中、拖拽、resize、连接预览”看 `interaction`
- “正式图状态怎么改”不要在这里找，回 `graph`

补充说明：

- 右键菜单能力已经迁到 `@leafergraph/context-menu`
- 主包 `interaction` 目录当前只保留图交互宿主，不再承载公共菜单实现

## 4. 四条关键主链

## 4.1 节点主链

节点相关能力当前大致按这条链流动：

```text
NodeRegistry
  -> node_shell_host / node_layout / node_shell
  -> node_host
  -> graph_scene_host
  -> graph_mutation_host / graph_restore_host
  -> graph_scene_runtime_host
  -> graph_api_host
```

更细一点：

- `NodeRegistry`
  - 负责“某个类型是否合法、有没有注册”
- `node_shell_host`
  - 负责“这个节点壳该长什么样”
- `node_host`
  - 负责“把节点视图 mount/refresh/unmount 到 scene”
- `graph_mutation_host`
  - 负责“节点创建/更新/移动/缩放/删除”
- `graph_scene_runtime_host`
  - 负责“把节点刷新和正式图变更收敛成统一壳面”

## 4.2 连线主链

```text
graph_mutation_host
  -> link_host
  -> graph_scene_host
  -> graph_scene_runtime_host
  -> graph_api_host
```

如果再带上执行动画：

```text
node_runtime_host link propagation
  -> link_data_flow_animation_host
  -> renderFrame()
```

也就是说：

- 正式连线路径刷新走 `link_host`
- 数据流动画走 `link_data_flow_animation_host`
- 两条链不要混写

## 4.3 Widget 主链

```text
graph_widget_runtime_host
  -> widget_registry
  -> widget_host
  -> node_host
  -> graph_scene_runtime_host
  -> graph_api_host
```

编辑相关的旁路是：

```text
theme_host
  -> widget_editing_manager
  -> widget_editing_context
  -> renderer context
```

`widget.commit` 这条链还会经过：

```text
widget_host
  -> sceneRuntimeHost.setNodeWidgetValue(...)
  -> interactionCommitSource.emit(...)
```

## 4.4 交互主链

```text
interaction_host
  -> interaction_runtime_host
  -> scene_runtime_host / node_runtime_host / view_host
  -> interaction_commit_source
  -> graph_api_host subscribeInteractionCommit(...)
```

它的重点是：

- 交互层负责指针和局部预览
- 正式图提交仍然回到 `graph` 和 `api`

## 5. 关键宿主怎么看

如果你只想快速判断“这个文件是装配型、刷新型、执行型还是 facade”，可以按下面这张表：

| 宿主 | 类型 | 主要作用 |
| --- | --- | --- |
| `graph_runtime_assembly` | 装配型 | 串总装配顺序 |
| `graph_scene_runtime_assembly` | 装配型 | 串场景宿主顺序 |
| `graph_bootstrap_host` | 启动型 | 装内建、装模块、装插件、恢复文档 |
| `graph_scene_runtime_host` | 场景 facade | 统一节点/连线刷新和正式图变更 |
| `graph_mutation_host` | 变更型 | 正式图增删改移 |
| `graph_restore_host` | 恢复型 | 整图恢复 |
| `graph_execution_runtime_host` | 执行型 | 图级执行状态机和 timer |
| `node_runtime_host` | 执行/节点态 | 节点执行、输入输出、状态变化 |
| `widget_host` | Widget 运行型 | renderer 调度 |
| `interaction_host` | 交互型 | 指针事件与交互生命周期 |
| `graph_api_host` | facade | 对外稳定 API |

## 7. 与其它包的协作边界

## 7.1 与 `@leafergraph/node`

`@leafergraph/node` 当前负责：

- `NodeDefinition`
- `NodeModule`
- `NodeRegistry`
- `GraphDocument`
- Widget 定义读取和序列化工具

`leafergraph` 不应该重新定义这些真源。

主包应该做的是：

- 读取这些定义
- 把它们恢复成运行时场景
- 提供执行、交互、渲染和 facade

## 7.2 与外部宿主

外部宿主当前负责：

- 页面壳层与工作区 UI
- 命令组织与菜单语义
- session / authority 一类外围协议接线
- bundle loader 与本地装配入口
- 远端 runtime 的额外接线

主包不应该回流这些能力：

- `LeaferGraphEditorBundleBridge`
- bundle manifest / bundle catalog
- authority transport / OpenRPC envelope
- workspace 级菜单和面板状态

如果你看到主包开始感知这些概念，通常就说明边界开始松了。

## 8. 当前维护约束

做主包改动时，建议始终检查这几个问题：

### 8.1 这个改动是不是放错层了

先问：

1. 这是模型定义吗
2. 这是主包运行时吗
3. 这是 editor 壳层吗

如果是第 1 种，回 `@leafergraph/node`。  
如果是第 3 种，不要塞回主包。

### 8.2 这个宿主是不是承担太多职责了

当前主包已经用“多个 host + facade”的方式拆层。  
如果一个文件开始同时做：

- 场景装配
- 正式图变更
- DOM 交互
- 外部协议适配

那通常就该继续拆。

### 8.3 这个问题是刷新链还是执行链

主包里这两条链经常被混淆：

- 刷新链
  - 节点壳、连线、Widget、渲染请求
- 执行链
  - 图级运行、节点执行、传播、反馈、动画

排查前先分清是哪一条，能少走很多弯路。
