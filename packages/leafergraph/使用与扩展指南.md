# 使用与扩展指南

这份文档面向 `leafergraph` 的使用者。

目标不是解释全部内部实现，而是把当前主包真正稳定可用的使用路径讲清楚：

- 怎么创建图实例
- 怎么输入和同步正式文档
- 怎么注册节点、模块、插件和 Widget
- 怎么执行图、订阅反馈、处理局部更新
- 哪些能力属于主包，哪些能力属于 `@leafergraph/node` 或 editor
- 执行相关类型应该从 `@leafergraph/execution` 还是 `leafergraph` 获取

## 包边界速记

- `@leafergraph/node`
  - 节点模型、模块、注册表、文档真源
- `@leafergraph/execution`
  - 执行上下文、`onAction(...)` 传播语义、图级 `play / step / stop`、执行反馈
- `@leafergraph/basic-kit`
  - 默认系统节点、基础 widgets、显式安装 plugin
- `leafergraph`
  - Leafer 宿主、节点/连线/Widget 场景、交互、`node.state` 投影、图 façade

执行相关类型的推荐导入方式：

```ts
import type {
  LeaferGraphExecutionContext,
  LeaferGraphActionExecutionOptions,
  LeaferGraphTimerRuntimePayload
} from "@leafergraph/execution";

import type { RuntimeFeedbackEvent } from "leafergraph";
```

也就是说：

- 纯执行类型优先从 `@leafergraph/execution` 导入
- 宿主级 superset 反馈 `RuntimeFeedbackEvent` 继续从 `leafergraph` 导入

## 1. 创建图实例

主入口只有两个：

- `new LeaferGraph(container, options?)`
- `createLeaferGraph(container, options?)`

推荐使用 `createLeaferGraph(...)`。

```ts
import { createLeaferGraph, type GraphDocument } from "leafergraph";

const container = document.getElementById("app");
if (!container) {
  throw new Error("缺少挂载容器 #app");
}

const initialDocument: GraphDocument = {
  documentId: "guide-document",
  revision: 1,
  appKind: "leafergraph-local",
  nodes: [],
  links: []
};

const graph = createLeaferGraph(container, {
  fill: "#0b1220",
  document: initialDocument
});

await graph.ready;
```

### `ready` 的意义

`graph.ready` 会等待启动装配链完成，主要包含：

- `modules` 安装
- `plugins` 安装
- 初始 `document` 恢复

如果你在外部要依赖这些结果，优先 `await graph.ready`。

### 1.1 显式安装默认内容包

主包不再默认内装基础 widgets 和系统节点。

如果你需要 `system/on-play`、`system/timer` 或基础 widgets，请显式安装 `@leafergraph/basic-kit`：

```ts
import { leaferGraphBasicKitPlugin } from "@leafergraph/basic-kit";

const graph = createLeaferGraph(container, {
  fill: "#0b1220",
  document: initialDocument,
  plugins: [leaferGraphBasicKitPlugin]
});
```

## 2. `LeaferGraphOptions` 里能配什么

主包初始化目前只保留正式宿主入口：

```ts
interface LeaferGraphOptions {
  fill?: string;
  document?: GraphDocument;
  modules?: NodeModule[];
  plugins?: LeaferGraphNodePlugin[];
  themePreset?: string;
  themeMode?: "light" | "dark";
  widgetEditing?: LeaferGraphWidgetEditingOptions;
  linkPropagationAnimation?: "performance" | "balanced" | "expressive" | false;
}
```

这里的边界要特别注意：

- `document` 是正式图输入
- `modules` 和 `plugins` 是扩展入口
- `NodeModule` 来自 `@leafergraph/node`
- `LeaferGraphNodePlugin` 来自 `leafergraph`
- `themePreset` / `themeMode` 的真源来自 `@leafergraph/theme`

## 3. 文档输入与同步

主包当前有三条常用文档同步路径。

### 3.1 `replaceGraphDocument(...)`

适合：

- 初始化整图
- authority full snapshot
- 明确知道需要整图替换的场景

```ts
graph.replaceGraphDocument(nextDocument);
```

语义是：

- 清理当前运行时图状态
- 按新文档重新恢复节点、连线、Widget 和视图
- 这条路径最重，但最稳

### 3.2 `applyGraphOperation(...)`

适合：

- 本地命令
- UI 交互提交
- 后端回放单条正式操作

```ts
const result = graph.applyGraphOperation({
  operationId: "op:node.move:1",
  timestamp: Date.now(),
  source: "example",
  type: "node.move",
  nodeId: "node-1",
  input: { x: 240, y: 160 }
});
```

返回值是 `GraphOperationApplyResult`，会告诉你：

- `accepted`
- `changed`
- `affectedNodeIds`
- `affectedLinkIds`
- `reason`

### 3.3 `applyGraphDocumentDiff(...)`

适合：

- authority diff
- session 层已合成 `nextDocument` 的增量投影

```ts
const projection = graph.applyGraphDocumentDiff(diff, nextDocument);

if (projection.requiresFullReplace) {
  graph.replaceGraphDocument(nextDocument);
}
```

这条路径和 `applyGraphDocumentDiffToDocument(...)` 的区别是：

- `applyGraphDocumentDiffToDocument(...)`
  - 纯文档工具
  - 不依赖图实例
  - 适合实例外先推导文档
- `graph.applyGraphDocumentDiff(...)`
  - 场景级增量投影
  - 会尝试局部刷新节点、连线和 Widget
  - 失败时要求回退到 full replace

## 4. 创建、更新和删除节点/连线

### 4.1 创建节点

```ts
graph.createNode({
  id: "node-1",
  type: "example/hello",
  x: 120,
  y: 80,
  width: 280,
  height: 160,
  title: "Hello",
  properties: {
    subtitle: "Created from API"
  }
});
```

`LeaferGraphCreateNodeInput` 现在仍保留扁平的 `x / y / width / height` 字段，目的是让交互和页面层调用更直接。

### 4.2 更新节点

```ts
graph.updateNode("node-1", {
  title: "Hello Updated",
  x: 180,
  y: 120,
  properties: {
    subtitle: "Updated by API"
  }
});
```

### 4.3 移动与缩放

```ts
graph.moveNode("node-1", { x: 240, y: 140 });
graph.resizeNode("node-1", { width: 320, height: 200 });
```

### 4.4 创建与删除连线

```ts
graph.createLink({
  id: "link-1",
  source: { nodeId: "node-1", slot: 0 },
  target: { nodeId: "node-2", slot: 0 }
});

graph.removeLink("link-1");
```

## 5. 节点扩展：`registerNode(...)`、`installModule(...)`、`use(plugin)`

这三条路径都是扩节点，但职责不同。

### 5.1 `registerNode(...)`

适合单个节点定义。

```ts
import type { NodeDefinition } from "@leafergraph/node";

const helloNode: NodeDefinition = {
  type: "example/hello",
  title: "Hello",
  category: "Examples",
  outputs: [{ name: "Text", type: "string" }]
};

graph.registerNode(helloNode, { overwrite: true });
```

### 5.2 `installModule(...)`

适合一组节点定义共用同一命名空间和分组。

```ts
import type { NodeModule } from "@leafergraph/node";

const exampleModule: NodeModule = {
  scope: {
    namespace: "example",
    group: "Examples"
  },
  nodes: [
    {
      type: "hello",
      title: "Hello"
    }
  ]
};

graph.installModule(exampleModule, { overwrite: true });
```

模块安装时，`@leafergraph/node` 会负责：

- 解析 `scope.namespace`
- 计算最终节点类型
- 在缺失 `category` 时继承 `scope.group`

### 5.3 `use(plugin)`

适合完整扩展包。

```ts
import type { LeaferGraphNodePlugin } from "leafergraph";

const examplePlugin: LeaferGraphNodePlugin = {
  name: "example/plugin",
  install(ctx) {
    ctx.registerNode(
      {
        type: "example/from-plugin",
        title: "Plugin Node"
      },
      { overwrite: true }
    );
  }
};

await graph.use(examplePlugin);
```

插件上下文 `LeaferGraphNodePluginContext` 当前能做这些事：

- `installModule(...)`
- `registerNode(...)`
- `registerWidget(...)`
- `hasNode(...)`
- `hasWidget(...)`
- `getNode(...)`
- `getWidget(...)`
- `listNodes()`
- `listWidgets()`

### 推荐顺序

如果你自己在外部组织扩展，推荐顺序是：

1. 先准备 Widget
2. 再安装 NodeModule
3. 最后恢复依赖这些类型的 `document`

如果是插件内部处理，推荐让 `install(ctx)` 自己按顺序完成。

### 5.4 `event` 输入与 `onAction(...)`

如果某条正式连线传播命中了目标输入槽位 `type === "event"`，主包当前采用这套稳定行为：

- 节点定义实现了 `onAction(...)` 时
  - 运行时会调用 `onAction(node, input.name, payload, options, api)`
- `options.executionContext`
  - 是当前这条执行链的正式 `LeaferGraphExecutionContext`
- `options.propagation`
  - 会带目标输入槽位索引 / 名称 / 类型
  - 也会带来源节点、来源输出槽位和连线 ID
- 如果目标输入不是 `event`
  - 仍然继续走 `onExecute(...)` 的数据流语义

这意味着 `events/*` 一类节点可以稳定依赖：

- `event` 输入
- `onAction(...)`
- `api.setOutputData(...)` 触发的正式传播链

当前阶段 `onTrigger(...)` 仍不作为主包 v1 的稳定依赖。

## 6. Widget 扩展

Widget 扩展走 `registerWidget(...)`，输入类型是 `LeaferGraphWidgetEntry`。

### 6.1 最小条目

```ts
import type { LeaferGraphWidgetEntry } from "leafergraph";

const exampleWidget: LeaferGraphWidgetEntry = {
  type: "example/status",
  title: "Status Widget",
  normalize(value) {
    return typeof value === "string" ? value : "idle";
  },
  serialize(value) {
    return typeof value === "string" ? value : "idle";
  },
  renderer(context) {
    const label = new context.ui.Text({
      text: String(context.value ?? "idle"),
      x: context.bounds.x,
      y: context.bounds.y
    });

    context.group.add(label);

    return {
      update(newValue) {
        label.text = String(newValue ?? "idle");
      },
      destroy() {
        label.remove();
      }
    };
  }
};

graph.registerWidget(exampleWidget, { overwrite: true });
```

### 6.2 推荐写法：生命周期对象

主包同时支持函数 renderer 和生命周期对象。复杂 Widget 更推荐生命周期对象：

```ts
import type {
  LeaferGraphWidgetEntry,
  LeaferGraphWidgetLifecycle
} from "leafergraph";

interface StatusState {
  text: import("leafer-ui").Text;
}

const lifecycle: LeaferGraphWidgetLifecycle<StatusState> = {
  mount(context) {
    const text = new context.ui.Text({
      x: context.bounds.x,
      y: context.bounds.y,
      text: String(context.value ?? "idle")
    });

    context.group.add(text);
    return { text };
  },
  update(state, _context, newValue) {
    state?.text && (state.text.text = String(newValue ?? "idle"));
  },
  destroy(state) {
    state?.text?.remove();
  }
};

const widget: LeaferGraphWidgetEntry = {
  type: "example/status",
  renderer: lifecycle
};
```

### 6.3 常用 Widget helper

主包公开了几组常用 helper：

- `createWidgetHitArea(...)`
- `createWidgetLabel(...)`
- `createWidgetSurface(...)`
- `createWidgetValueText(...)`
- `bindPressWidgetInteraction(...)`
- `bindLinearWidgetDrag(...)`

它们适合复用主包现有的 Widget 手感和命中策略。

## 7. 运行控制

### 7.1 图级运行

```ts
graph.play();
graph.step();
graph.stop();
```

语义分别是：

- `play()`
  - 从图中全部 `system/on-play` 启动事件节点开始图级运行
- `step()`
  - 单步推进一次图级运行
- `stop()`
  - 停止当前图级运行

### 7.2 节点级运行

```ts
graph.playFromNode("node-1");
```

这条路径适合本地调试一个节点或一小段执行链。

### 7.3 `system/on-play`（Start Event）和 `system/timer`

这两个系统节点不再由主包默认内装。
如果你要使用它们，请先显式安装 `@leafergraph/basic-kit`，或直接从 `@leafergraph/basic-kit/node` 消费定义。

如果你要组织一条最小执行链，可以这样理解：

- `system/on-play`
  - 用于图级启动事件
  - 输出当前执行上下文
- `system/timer`
  - 需要上游 `Start`
  - 图级运行时可注册定时循环
  - 节点级运行只会立刻执行一次，不会进入图级循环

## 8. 订阅反馈

主包提供这几条订阅链：

```ts
const offNodeExecution = graph.subscribeNodeExecution((event) => {
  console.log("node execution", event.nodeId, event.state.status);
});

const offGraphExecution = graph.subscribeGraphExecution((event) => {
  console.log("graph execution", event.type, event.state.status);
});

const offNodeState = graph.subscribeNodeState((event) => {
  console.log("node state", event.nodeId, event.reason);
});

const offRuntime = graph.subscribeRuntimeFeedback((event) => {
  console.log("runtime feedback", event.type);
});
```

### 各自适合什么场景

| API | 适合什么 |
| --- | --- |
| `subscribeNodeExecution(...)` | 时间线、运行日志、失败面板 |
| `subscribeGraphExecution(...)` | 图级状态栏、运行控制 UI |
| `subscribeNodeState(...)` | 检查器、节点状态同步 |
| `subscribeRuntimeFeedback(...)` | 外部 runtime 适配层或统一调试入口 |

## 9. 投影外部 runtime feedback

如果你已经有外部执行反馈，可以直接投影回主包：

```ts
graph.projectRuntimeFeedback({
  type: "graph.execution",
  event: {
    type: "started",
    state: {
      status: "running",
      runId: "remote-run-1",
      queueSize: 1,
      stepCount: 0
    },
    runId: "remote-run-1",
    source: "graph-play",
    timestamp: Date.now()
  }
});
```

这条 API 的定位是：

- 复用主包已有的节点、连线、动画和状态投影链
- 不要求外部系统直接操作 Leafer 场景
- 让外部 runtime 和本地主包共用一套 UI 表达

它不是 authority transport，也不替代正式文档同步。

## 10. 常用工具和交互能力

### 10.1 端口命中与连线辅助

```ts
const port = graph.resolveConnectionPort("node-1", "output", 0);
const hit = graph.resolveConnectionPortAtPoint({ x: 100, y: 200 }, "input");
```

这组 API 适合：

- 自定义连线交互
- 调试端口命中框
- 外部交互桥接

### 10.2 菜单

```ts
import { createLeaferContextMenu } from "@leafergraph/context-menu";
```

右键菜单已经不再由 `leafergraph` 主包承载，而是改为独立的 Leafer-only 包。

推荐做法是：

- 图实例继续由 `leafergraph` 创建和管理
- 菜单系统直接使用 `@leafergraph/context-menu`
- 画布、节点、连线的 target 绑定由外层页面或 demo 在 `graph.ready` 后完成

这样菜单 schema、子菜单状态和目标分类会和宿主图运行时解耦，主包只继续负责图本身。

### 10.3 主题

```ts
graph.setThemeMode("dark");
graph.setThemeMode("light");
```

初始化时如果要指定命名主题 preset，可在构造图实例时直接传：

```ts
const graph = createLeaferGraph(container, {
  plugins: [leaferGraphBasicKitPlugin],
  themePreset: "default",
  themeMode: "dark"
});
```

主题切换会触发：

- 节点壳刷新
- 连线刷新
- Widget 主题刷新

但不会替你处理页面级 CSS 主题。

视觉主题 token、默认 preset 和 preset 注册表现在统一来自 `@leafergraph/theme`。

## 11. 一个完整的小示例

下面是“创建图 + 注册节点 + 创建实例 + 运行”的最小闭环：

```ts
import { leaferGraphBasicKitPlugin } from "@leafergraph/basic-kit";
import type { NodeDefinition } from "@leafergraph/node";
import { createLeaferGraph, type GraphDocument } from "leafergraph";

const container = document.getElementById("app");
if (!container) {
  throw new Error("缺少 #app");
}

const onPlayDocument: GraphDocument = {
  documentId: "example-document",
  revision: 1,
  appKind: "leafergraph-local",
  nodes: [
      {
        id: "entry",
        type: "system/on-play",
        title: "Start Event",
        layout: { x: 80, y: 80 }
      },
    {
      id: "hello",
      type: "example/hello",
      title: "Hello",
      layout: { x: 320, y: 80 }
    }
  ],
  links: [
    {
      id: "entry->hello",
      source: { nodeId: "entry", slot: 0 },
      target: { nodeId: "hello", slot: 0 }
    }
  ]
};

const helloNode: NodeDefinition = {
  type: "example/hello",
  title: "Hello",
  category: "Examples",
  inputs: [{ name: "Event", type: "event" }],
  outputs: [{ name: "Text", type: "string" }],
  onExecute(node, _context, api) {
    node.title = "Hello Executed";
    api?.setOutputData(0, "hello");
  }
};

const graph = createLeaferGraph(container, {
  document: onPlayDocument,
  plugins: [leaferGraphBasicKitPlugin]
});

await graph.ready;
graph.registerNode(helloNode, { overwrite: true });
graph.play();
```

更真实的做法通常是先把节点定义放进 `modules` 或 `plugins`，再把文档恢复进来。

## 12. 常见误区

### 不要把 editor bundle loader 当成 `leafergraph` 公共 API

`LeaferGraphEditorBundleBridge`、bundle manifest 和本地 bundle 管理面板都属于外部宿主接入层，不是主包公共契约。

### 不要把 `@leafergraph/node` 和 `leafergraph` 写混

- `NodeDefinition`、`NodeModule`、`GraphDocument` 模型结构真源在 `@leafergraph/node`
- 图宿主、场景刷新、执行反馈、Widget 渲染宿主在 `leafergraph`

### 不要把 authority / session 传输层当成主包前提

主包可以独立使用。  
authority、OpenRPC、session、transport 是 editor 或更外层系统的装配问题。

### 不要把 `replaceGraphDocument(...)` 和 `applyGraphOperation(...)` 混成一回事

- `replaceGraphDocument(...)` 是整图恢复
- `applyGraphOperation(...)` 是正式单条操作
- `applyGraphDocumentDiff(...)` 是增量投影

### 不要假设节点或 Widget 一定来自 editor

主包本身支持：

- 直接 `registerNode(...)`
- 直接 `installModule(...)`
- 直接 `registerWidget(...)`
- 直接 `use(plugin)`

你可以完全绕开 editor 自己装配。
