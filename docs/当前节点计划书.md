这份计划书不再讨论“节点系统要不要做”，而是明确回答下面四个问题：

1. 当前节点体系已经真正落地了哪些能力
2. 当前哪些内容虽然可见，但本质上仍然属于 demo 或过渡实现
3. 下一步最应该先补什么，而不是继续横向加功能

- **当前项目已经完成“节点 SDK、正式图输入、唯一 Widget 注册中心、正式图 API、基础编辑交互、现代化内建 Widget、editor 本地 bundle 装载、外部模板工程”的第一阶段，但后端权威化基础、统一操作模型、运行反馈模型和外部 runtime 对接仍未完成。**

### 2.1 当前目标分层

在“后端权威化 + 多应用可接入编辑器”方向下，后续文档和实现都应围绕下面六类核心概念收敛：

1. `Document Model`
   - 即 `GraphDocument`
   - 表示统一图文档、节点 / 连线 / 布局 / 属性快照等可持久化、可恢复、可由后端持有的正式数据
2. `Operation Model`
   - 即 `GraphOperation`
   - 表示统一图变更、创建、删除、移动、连线、批量操作、撤销重做等正式操作语义
3. `Capability Model`
   - 即 `CapabilityProfile`
   - 表示某类外部应用 / runtime 支持哪些建模能力、运行能力和编辑能力
4. `Interaction View Model`
   - 表示选区、拖拽会话、重连会话、临时高亮、视口等前端瞬时交互态
5. `Runtime Feedback Model`
   - 即 `RuntimeFeedbackEvent`
   - 由 `RuntimeAdapter` 表达本地执行器、后端执行器或外部 runtime 的事件来源
   - 表示节点状态、链路流动、日志、错误、执行链和检查面板所需事件
6. `Adapter Model`
   - 即 `AppAdapter`
   - 表示 `Canonical Model <-> External App Model` 的映射和接入边界

职责约定：

- `Document Model`、`Operation Model` 与 `Capability Model` 默认以后端为主，前端负责编辑、提交和消费确认结果，不长期并列维护事实真源
- `Interaction View Model` 默认以前端为主，不直接等价于正式图文档
- `Runtime Feedback Model` 默认由后端或外部 runtime 提供，前端负责消费、投影和调试显示
- `AppAdapter` 与 `RuntimeAdapter` 停留在适配层，外部应用通过 adapter 接入，而不是反向塑造核心模型
- 现有前端执行链继续保留，但定位收敛为本地验证、演示和离线回退能力，不再作为唯一正式执行主路径

## 6. 接下来必须遵守的原则

### 6.1 先收敛节点输入模型，再继续横向加壳层功能

当前最需要克制的，不是再补几个面板或再扩几个 demo 节点，而是避免继续把 `packages/node/src/demo.ts` 里的演示输入当作事实标准。

接下来新增能力时，优先级应该固定为：

1. 正式图数据入口与持久化边界
2. 主包宿主拆层
3. editor 命令总线与历史记录
4. 连接与执行闭环

### 6.2 `@leafergraph/node` 继续保持宿主无关，只承接 Node SDK

这个包接下来仍应只负责：

- `NodeDefinition / WidgetDefinition` 等数据契约
- `NodeRegistry`、模块安装与作用域解析
- `createNodeState()`、`configureNode()`、`serializeNode()`、`createNodeApi()`
- 生命周期类型与宿主无关的约束协议

这个包不应该再继续吸收：

- Leafer 图元与渲染宿主
- DOM bridge、IIFE bundle 协议
- editor 工作分页、面板状态、命令面

### 6.3 `leafergraph` 主包继续做唯一运行时宿主，不回退到双轨结构

主包当前已经形成了比较清晰的边界，后续应该继续强化，而不是反向混乱：

- 继续由主包唯一持有 `NodeRegistry + LeaferGraphWidgetRegistry`
- 继续由主包承接节点、连线、Widget、编辑宿主与主题宿主
- 继续由主包承接缺失节点 / 缺失 Widget fallback、右键菜单基础设施、正式图 API

同时必须避免重新出现这些回退：

- Widget definition 和 renderer 再次分开注册
- `NodeModule` 再次混入 Widget 协议
- demo 初始化逻辑重新塞回主包公共 API

### 6.4 `editor` 继续负责工作分页、本地 bundle 装载和命令接线

当前 editor 的职责已经比早期更清楚，后续应继续保持：

- `App.tsx` 负责工作分页、开始 / 停止渲染和本地 bundle 面板
- `loader/runtime.ts` 负责 `LeaferGraphRuntime`、`LeaferGraphEditorBundleBridge` 与 manifest 校验
- `GraphViewport.tsx` 负责挂载主包实例、菜单、快捷键和主题同步
- `node_commands.ts`、`canvas_commands.ts` 负责最小命令面

其中需要特别强调：

- IIFE bundle bridge 属于 editor 接入层协议，不进入主包公共 API
- editor 可以控制装载与命令，但不应重复实现图模型和渲染宿主

### 6.5 外部生态优先走标准路径，不把 sandbox 协议误当主包协议

当前已经存在两条对外路径，但语义不同，后续要坚持区分：

1. 标准包接入：
   - 节点通过 `NodeModule`
   - Widget 通过 `ctx.registerWidget(entry)`
2. editor sandbox 接入：
   - 通过本地 dist IIFE
   - 通过 `LeaferGraphEditorBundleBridge.registerBundle(...)`

前者是长期生态接口，后者只是 editor 的本地调试 / 演示装配协议。

   - 不再把 editor 的执行入口写成唯一正式运行入口
2. 建立运行时边界

   - 用 `RuntimeAdapter` 表达本地执行器、后端执行器和外部 runtime 的统一接入口

3. 收敛正式运行执行路径

   - 前端负责发起 `GraphOperation`、消费事件、刷新显示，不假定执行必须发生在浏览器内

4. 为后续多应用 adapter 预留边界
   - 核心库只定义文档、操作、能力和反馈抽象，不直接写死某个外部应用协议
   - 把外部模型映射、运行时桥接和可视反馈映射留给 `AppAdapter` / `RuntimeAdapter`
5. 回接执行检查面板
   - 在不扩大本地执行器职责的前提下，继续完善右侧信息面板、时间线和最近失败列表
   - 保证相同 UI 能消费本地验证事件与外部 runtime 事件

完成标准：

- 文档上已清楚区分“本地验证执行”“正式运行执行”“外部 runtime 同步显示”
- `onConnectionsChange(...)` 已作为正式宿主生命周期稳定成立
- `onExecute(...)` 被明确标记为已有本地原型，而不是未开始
- `RuntimeFeedbackEvent`、`CapabilityProfile` 与 `AppAdapter` / `RuntimeAdapter` 已成为后续实现边界
- 运行反馈 UI 的输入语义不再依赖“浏览器内必须存在唯一调度器”

当前进展：

- `onConnectionsChange(...)` 已完成恢复路径补发与宿主级异常隔离
- 主包已补 `executeNode(nodeId, context?)` 正式入口
- `setOutputData(...)` 已能沿现有 `graphLinks` 写回下游 `inputValues`，并按最小递归策略继续触发下游 `onExecute(...)`
- editor 已接入 `node.execute` 命令和节点菜单入口
- 模板工程已补 `Counter Source -> Display` 执行样例
- 节点执行反馈已接入主包正式 runtime 状态，节点信号灯会根据 `idle / running / success / error` 切换颜色，失败节点会在主画布上直接显示节点级错误文案
- 主包已补节点执行事件订阅口，editor 的 timeline 已从“命令结果”切到“真实执行链事件”，下游传播触发的节点也会进入时间线
- 主包已补节点检查快照读取口，editor 右侧已整理成正式信息面板，可查看当前焦点节点的内部数据、properties 和输入输出 IO 值
- editor 已新增最小运行反馈信息面板，会跟随当前主选中节点或最近执行节点显示执行状态、最近命令、最近执行链、最近失败列表和节点内部状态
- editor 已落地 `Document Session` authority 骨架：同一套会话接口支持 `loopback` 与 `remote-mock` 两种模式，命令链已能处理 `pending / confirmed / rejected`，且 history 仅在 confirmed 后入栈
- `replaceDocument(...)` 已收敛到 session authority 入口，bundle 切换与整图替换不再走散落直写路径
- editor 已新增 `RuntimeFeedbackInlet` 外部反馈入口，本地执行反馈与外部反馈统一走 `RuntimeFeedbackEvent` 处理链，UI 不新增私有反馈分叉
- editor 已开始抽真实 authority 接入边界：新增 `RemoteAuthorityClient` 协议、`remote-client` session / binding 模式，以及 `GraphViewport` 对 authoritative `GraphDocument` 回推的投影路径；但 authority-first 下的完整命令交互仍未收敛
- editor 的核心 create / remove / resize / link.create / link.reconnect 命令已开始适配 authority-first 语义：pending 状态不再被误判为失败，最小历史负载与目标尺寸 / 目标端点信息可以保留下来；但真实 transport、GraphViewport 级交互回归和外部 runtime feedback 联调仍未完成
- 后端联调前，当前已补一条可由 `node` 驱动的 JS demo backend 测试夹具，可单独验证 `getDocument / submitOperation / replaceDocument` 最小 authority 链路
- 已补最小回归用例覆盖 authority 流转与反馈入口：`pending -> confirmed / rejected`、`replaceDocument` 取消 pending、history 在 pending 确认后入栈、manual feedback inlet 注入分发
- 上述能力当前仍主要服务本地验证路径，后端权威化执行模型、外部 runtime 反馈桥和 Dora runtime adapter 仍未开始

运行时边界：

- 核心库只定义 `GraphDocument`、`GraphOperation`、`CapabilityProfile`、`RuntimeFeedbackEvent`、`AppAdapter` 和 `RuntimeAdapter` 这类抽象边界，不预设执行一定发生在浏览器内
- 执行状态展示要能接受后端或外部 runtime 事件源，而不是只读取内部调度器快照
- 链路高亮、节点灯、最近失败列表、检查面板要统一消费通用反馈事件
- 本地执行链继续保留，但属于验证 / 演示 / 离线回退能力

#### D-5：多应用接入方向

定位：

- `LeaferGraph` 在该方向中的角色是“面向多种外部应用 / runtime 的可视化流程编辑器 + runtime visualizer”
- 后端持有统一 `Canonical GraphDocument`、`Canonical GraphOperation` 与 `CapabilityProfile`
- 前端不需要为每种外部应用重写一套编辑器

##### D-5.1：通用 App Adapter 架构

第一阶段只要求文档上固定下面六类概念接口：

- `GraphDocument`：统一图文档抽象
- `GraphOperation`：统一图操作抽象
- `CapabilityProfile`：某类外部应用 / runtime 支持哪些建模、运行与编辑能力
- `RuntimeFeedbackEvent`：外部 runtime 回流到 UI 的统一事件抽象
- `AppAdapter`：`Canonical Model <-> External App Model` 的统一适配接口
- `AdapterBinding`：图文档与目标应用 / adapter 的绑定元数据

适配层职责：

- 后端通过 `Adapter Registry` / `Adapter Router` 选择目标 adapter
- 新应用接入优先新增 model mapper、capability resolver、runtime event normalizer
- `CapabilityProfile` 负责把“某个应用支持哪些能力”投影回前端，而不是让前端写死产品差异
- 前端继续消费统一 `GraphDocument` 与 `RuntimeFeedbackEvent`，不因新增应用重写编辑器主流程

##### D-5.2：Dora 作为首个重点适配对象

定位：

- Dora 是首个重点落地的 adapter，对当前路线有优先级，但不再主导通用术语
- `LeaferGraph` 在 Dora 方向中的具体角色仍然是 `Dora dataflow editor + runtime visualizer`
- 不要求把 Dora runtime 搬进前端
- 不要求当前本地节点协议与 Dora 节点协议完全等价

后续 Dora adapter 职责：

- `LeaferGraphData <-> Dora dataflow` 映射
- Dora runtime 事件 -> `RuntimeFeedbackEvent` 映射
- Dora 节点 / 边 / 状态 -> 前端高亮、链路流动和检查面板映射

非目标：

- 在核心库中硬编码 Dora wire format、CLI 参数或进程管理细节
- 为了 Dora 牺牲 runtime-agnostic 的主包边界
- 让 Dora 成为 `GraphDocument`、`GraphOperation` 或 `CapabilityProfile` 的命名来源
- 要求第一阶段就让本地执行链与 Dora 执行语义完全一致

## 12. 风险与控制

### 风险 1：继续在 demo 类型上堆字段

后果：

- `packages/node/src/demo.ts` 会继续占据 SDK 语义中心
- 正式图输入和持久化恢复模型会越来越难独立出来

控制方式：

- 新需求优先落到正式图数据和恢复模型
- demo 输入只在 editor / template 适配层继续扩展
- 文档中显式区分“演示输入”和“正式输入”

### 风险 2：主包继续变成“所有事情的大文件”

后果：

- 节点、连线、Widget、编辑宿主越来越难独立演进
- `index.ts` 会重新变成高耦合汇总文件

控制方式：

- 优先推进宿主拆层，不把新能力继续堆进组装层
- 拆层时明确“节点宿主 / 连线宿主 / Widget 宿主 / 编辑宿主”边界

### 风险 3：把最小命令控制器和本地 bundle loader 误判为“产品化 editor 已完成”

后果：

- 会高估当前系统成熟度
- 延后命令总线、历史记录、执行闭环这些真正核心工作

控制方式：

- 明确当前 editor 仍是工作区壳层与调试壳，不是产品化完整编辑器
- 不把“能加载 bundle、能点击菜单”误判为“命令系统已完成”

### 风险 4：外部 bundle 协议和主包公共 API 混淆

后果：

- editor 本地调试协议会反向污染主包
- 外部作者会搞不清楚该用 `NodeModule`、`registerWidget(entry)` 还是 bundle bridge

控制方式：

- 明确 IIFE bundle bridge 仅属于 editor 接入层
- 主包文档继续把标准包接入作为长期主路径

### 风险 5：外部 Widget 作者各自发明一套主题与交互范式

后果：

- 外部 widgets 的观感、焦点行为和编辑体验会迅速分裂
- 主包已经提供的 `theme / editing / lifecycle` 能力很难被复用

控制方式：

- 继续沉淀标准 Widget 作者模板
- 补清楚主题 token、编辑宿主和焦点绑定的推荐写法
- 尽量让内建 widgets 成为外部作者可直接模仿的样板

### 风险 6：继续把前端本地状态当成事实真源

后果：

- 后端权威化会在后续阶段被迫重做
- 本地交互态、正式文档态和运行反馈态会继续混在一起

控制方式：

- 文档上明确 `GraphDocument`、`GraphOperation` 与 `CapabilityProfile` 默认以后端为主
- 前端只长期持有交互态和渲染态，不把它们误写成正式文档事实

### 风险 7：把命令总线只做成前端快捷键分发层

后果：

- 命令总线无法演进为后端确认、补丁回放和重建友好的正式操作模型
- 撤销重做、协同和运行时回放会越来越难接

控制方式：

- 文档上把 `GraphOperation` 作为长期正式抽象
- 后续命令设计优先考虑是否能映射到后端确认与回放，而不只服务本地 UI 入口

### 风险 8：缺少统一 `CapabilityProfile`，导致每接一个应用就重写一层编辑器假设

后果：

- 前端会把某个外部应用的能力约束硬编码到主流程里
- 新增应用时会不断复制条件分支，而不是复用统一能力投影

控制方式：

- 用 `CapabilityProfile` 收敛“支持哪些端口、节点、运行与交互能力”
- 前端根据 `CapabilityProfile` 调整 palette、表单、连线合法性和检查面板，而不是写死 Dora 或其他应用的产品语义

### 风险 9：把执行反馈继续绑定到本地执行器内部状态

后果：

- UI 无法稳定消费后端或 Dora runtime 事件
- 节点灯、链路高亮、失败列表和检查面板会被迫依赖浏览器内调度器

控制方式：

- 文档上正式引入 `RuntimeFeedbackEvent`
- 节点状态展示统一以反馈事件为输入，而不是默认只读取本地 scheduler

### 风险 10：过早把 Dora 协议硬编码进核心库

后果：

- 核心库 runtime-agnostic 边界被破坏
- 未来接入其他 runtime 或继续保留本地验证路径时会变得困难

控制方式：

- 把 Dora 专属内容限制在适配层
- 核心层只收敛 `GraphDocument`、`GraphOperation`、`CapabilityProfile`、`RuntimeFeedbackEvent`、`AppAdapter` 和 `RuntimeAdapter` 这类抽象

## 13. 阶段性验收标准

从当前 `Phase 1.9` 往下一阶段推进时，建议以这些条件作为阶段验收线：

- `AGENTS.md` 与本计划书都明确“后端权威化”为默认方向，前端不再被描述为与后端并列的长期事实真源
- 两份文档都不再把前端执行链描述为唯一正式执行模型，而是明确区分本地验证执行与正式运行执行
- 计划书内部不再同时出现“执行系统未开始”与“`onExecute(...)` 已接入完整链路”这类自相矛盾表述
- 文档中已正式出现 `GraphDocument`、`GraphOperation`、`CapabilityProfile`、`RuntimeFeedbackEvent`、`AppAdapter` 和 `RuntimeAdapter` 这些概念接口
- Dora 已被表述为首个重点 adapter，而不是通用模型来源；`LeaferGraphData <-> Dora dataflow` 映射和 runtime bridge 被限定在适配层
- 文档已明确“统一 Canonical Model + 多应用 adapter”主线，新接应用时优先新增 adapter，而不是重写主模型
- 主包的节点、连线、Widget、运行反馈和编辑宿主边界比当前更清楚，`index.ts` 更接近组装层
- editor 继续保持 IIFE bundle 仅属于接入层，不反向污染主包长期边界

## 14. 当前结论

当前节点系统已经明确跨过了“概念阶段”和“只有文档没有实现的阶段”，现在的真实状态更接近：

- `@leafergraph/node` 已成立
- 主包唯一 Widget 注册中心已成立
- 正式图输入、节点快照和最小编辑交互已成立
- 现代化基础 Widget 与编辑宿主已成立
- 本地验证执行与最小运行反馈已成立
- editor 工作分页与本地 bundle 装载链已成立
- 外部模板工程与 browser IIFE 接入样例已成立
- 但后端统一 `Canonical GraphDocument`、`Canonical GraphOperation`、`CapabilityProfile`、`RuntimeFeedbackEvent` 反馈桥和 `AppAdapter` 框架还没有完成

因此，接下来最重要的主线应该明确为：

**后端统一 `Canonical GraphDocument` 收敛 -> `Canonical GraphOperation` 收敛 -> `CapabilityProfile` 收敛 -> `RuntimeFeedbackEvent` 与 `RuntimeAdapter` 收敛 -> 建立 `AppAdapter` 框架并保持主包 runtime-agnostic -> Dora 作为首个重点 adapter 落地 -> 让后续其他应用低成本复制接入 -> 本地执行链收敛为验证 / 演示 / 离线回退能力**

最小自由连线、执行检查面板和模板作者体验仍要继续推进，但它们都应服从这条主线，这样后端化、Dora 接入和未来更多应用接入才不会把当前文档重新推翻。
