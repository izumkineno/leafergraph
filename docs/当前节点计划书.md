## 1. 计划目标

这份文档的目标不再是描述“理想中的节点系统”，而是回答下面四个当前问题：

1. 现在仓库里已经有哪些正式能力。
2. 哪些能力虽然可用，但仍属于过渡实现或 editor 接入层协议。
3. 接下来真正的主线应该是什么，而不是继续沿着旧 demo 口径发散。
4. 后端 authority、OpenRPC、runtime feedback 和 bundle/template 应该怎样放回同一张现状地图里。

当前统一口径如下：

- `@leafergraph/node` 负责节点定义、模块安装、节点序列化和 `GraphDocument` 基础模型。
- `leafergraph` 主包负责 Leafer 场景宿主、图变更 API、局部刷新、交互、运行时和 runtime feedback 投影。
- `leafergraph-editor` 负责工作区、命令总线、bundle 装载、authority session、菜单和 viewport 壳层。
- `openrpc/` 是 authority 协议真源目录；全仓通过 `LEAFERGRAPH_OPENRPC_ROOT` 或默认仓库根 `openrpc/` 读取。
- 后端权威化是长期主线；本地执行器和浏览器内 demo service 仍然保留，但默认视为验证、演示和离线回退路径。

## 2. 当前状态总览

### 2.1 包结构已经收敛到三层

当前工作区里和节点系统直接相关的核心目录已经稳定为：

| 层 | 当前目录 | 主要职责 |
| :--- | :--- | :--- |
| 节点模型层 | `packages/node` | `NodeDefinition`、`NodeModule`、`NodeRegistry`、`GraphDocument`、`CapabilityProfile` |
| 图运行时层 | `packages/leafergraph` | 场景装配、节点壳、连线、交互、执行反馈、图 API |
| 编辑器壳层 | `packages/editor` | viewport、命令、history、authority session、bundle loader、workspace UI |

已经不再推荐把不存在的旧包结构写成现状，例如旧的 node-sdk、node-templates、node-presets 命名。

### 2.2 authority 与协议真源已经收敛

当前 authority 协议链路的事实源如下：

- OpenRPC 真源目录：`openrpc/`
- 环境变量：`LEAFERGRAPH_OPENRPC_ROOT`
- 共享文档：`openrpc/authority.openrpc.json`
- editor 侧生成代码目录：`packages/editor/src/session/authority_openrpc/`
- Python authority 模板：`templates/backend/python-openrpc-authority-template`

这意味着：

- editor 不再以手写协议常量为唯一真源，而是已经开始消费共享 OpenRPC 生成物。
- 文档里不能再把旧模板共享 OpenRPC 目录写成现行目录。
- `rpc.discover`、`getDocument`、`submitOperation`、`replaceDocument`、`authority.document`、`authority.documentDiff`、`runtimeFeedback`、`frontendBundles.sync` 应统一按共享协议解释。

### 2.3 editor 现在同时有三条正式接入链

当前 editor 的主链已经不是“一个大页面 + 一些 demo 函数”，而是三条并行但边界明确的能力链：

1. 本地 bundle 装载链
   - `packages/editor/src/loader/runtime.ts`
   - `packages/editor/src/loader/types.ts`
   - `packages/editor/src/loader/persistence.ts`
2. authority 文档会话链
   - `packages/editor/src/session/graph_document_session.ts`
   - `packages/editor/src/session/graph_document_authority_transport.ts`
   - `packages/editor/src/session/authority_openrpc/`
3. viewport 与命令壳层链
   - `packages/editor/src/ui/viewport/View.tsx`
   - `packages/editor/src/commands/`
   - `packages/editor/src/menu/`
   - `packages/editor/src/shell/provider.tsx`

这三条链会在 `EditorProvider -> GraphViewport` 收口，但不应再被描述成“单个 GraphViewport 文件承接一切”。

## 3. 当前已落地的能力

### 3.1 `@leafergraph/node` 已经是正式模型入口

当前 `packages/node/src/index.ts` 已经对外稳定导出：

- `NodeDefinition`
- `NodeModule`
- `NodeRegistry`
- `createNodeApi`
- `installNodeModule`
- `GraphDocument`
- `CapabilityProfile`
- `GraphLink`

这说明节点系统已经不再依赖 demo 结构去描述正式模型。`packages/node/src/demo.ts` 仍然存在，但应视为历史兼容和接入层遗留，不是文档中心。

### 3.2 主包已经具备正式图 API 与局部刷新入口

`packages/leafergraph/src/index.ts` 当前已经对外暴露这些正式能力：

- 图文档整图替换：`replaceGraphDocument(...)`
- 图操作提交：`applyGraphOperation(...)`
- 图 diff 投影：`applyGraphDocumentDiff(...)`
- 外部运行反馈投影：`projectRuntimeFeedback(...)`
- 节点/连线/主题/交互查询与控制接口
- 右键菜单基础设施：`createLeaferGraphContextMenu(...)`

当前主包的核心装配链是：

1. `createLeaferGraphRuntimeAssembly(...)`
2. `LeaferGraphBootstrapHost`
3. `graph_scene_runtime_assembly.ts`
4. `LeaferGraphApiHost`

这条链已经比早期“所有东西都在入口大文件里”清楚得多。

### 3.3 文档同步链已经进入 authority-first 语义

`packages/editor/src/session/graph_document_session.ts` 当前已经具备：

- loopback session
- mock remote session
- remote authority session
- pending 操作队列
- confirmed / rejected 回执
- resync 机制
- `authority.documentDiff` 的文档增量合并

而 `packages/editor/src/ui/viewport/View.tsx` 会把 authority 文档投影分成两条路径：

- full snapshot 走 `graph.replaceGraphDocument(...)`
- diff 走 `graph.applyGraphDocumentDiff(...)`，失败再 full fallback

这已经不是最初的“每次都整图回填”状态。

### 3.4 runtime feedback 已经从本地执行器里抽象出来

当前主包已经有明确的 `RuntimeFeedbackEvent` 和 `RuntimeAdapter` 抽象：

- 本地执行由 `LeaferGraphLocalRuntimeAdapter` 归一成统一反馈事件
- 外部运行反馈由 `projectExternalRuntimeFeedback(...)` 分发到图、节点和连线宿主
- editor 侧通过 `runtimeFeedbackInlet` 订阅同一类反馈，而不是只依赖本地执行器内部状态

这为后端 runtime 或其它应用 runtime 回流 UI 打下了正式接口。

### 3.5 bundle/template 链已成立，但边界已经更清楚

当前模板与 editor 装载链的事实如下：

- 模板目录：`templates/misc/browser-node-widget-plugin-template`
- editor 本地 bundle 槽位：`demo`、`node`、`widget`
- authority 远端 bundle 格式：`script`、`node-json`、`demo-json`
- bundle 协议仅属于 editor 装载层，不进入主包公共 API

这里的重点不是“继续把 demo 当正式模型”，而是：

- 文档术语可以使用“蓝图 / 组件”，帮助读者理解职责
- 代码真实名字仍然是 `demo / widget / node`
- 两者需要显式映射，不能在文档里混成一团

## 4. 当前仍属于过渡实现的部分

### 4.1 本地 authority demo 仍然存在，但不再是唯一方向

当前 editor 仍保留：

- `src/demo/preview_remote_authority_bootstrap.ts`
- `src/demo/remote_authority_demo_service.ts`
- `src/demo/remote_authority_demo_worker.ts`

它们仍然有价值，但定位已经收敛为：

- 浏览器内预览
- 本地联调
- 无后端时的演示链

现行公开 authority demo 口径已经收口到 Python/OpenRPC 方向，默认入口是 `authority-python-host-demo.html`。

### 4.2 bundle 命名仍保留历史痕迹

当前最明显的过渡命名仍是：

- 代码槽位：`demo`
- 文档说明层：蓝图
- 代码槽位：`widget`
- 文档说明层：组件

这不是 bug，但文档必须持续提醒：

- 这是 editor 接入层协议的历史命名
- 不是主包领域模型本身的长期语义

### 4.3 editor 仍然是工作区壳层，不是产品化应用

当前 editor 已有不少正式能力，但仍应避免高估成熟度：

- 有命令总线，不代表事务系统和协同已完成
- 有 history，不代表所有交互都已纳入可逆操作
- 有 authority session，不代表所有远端链路都已完成产品化治理
- 有 bundle loader，不代表分发、版本兼容和发布渠道已经定型

### 4.4 后端权威化主线仍在继续推进

当前后端方向已经明确，但还不能说已经彻底完成：

- `GraphDocument`、`GraphOperation`、`GraphDocumentDiff`、`RuntimeFeedbackEvent` 已经有统一抽象
- Python OpenRPC authority 模板已经成为当前默认真实后端模板
- editor 已能消费 remote authority
- 但 `CapabilityProfile`、`AppAdapter`、跨应用适配与完整 runtime/authority 策略仍在继续收敛

## 5. 当前最关键的问题

### 5.1 文档口径必须持续保持“现状优先”

现在最大的风险已经不是“没有实现”，而是：

- 文档继续把旧阶段目标写成现状
- 路径和目录漂移后无人收口
- bundle、authority、runtime feedback、OpenRPC 这些新事实没有及时回写到总说明里

### 5.2 主包与 editor 的边界仍需继续守住

必须持续避免这两类回退：

- 把 editor 装载层协议重新塞回 `leafergraph` 主包
- 把主包内部运行时细节重新穿透到 editor UI 组件里

### 5.3 authority 文档同步与 runtime feedback 不能写混

当前已经出现两条不同通道：

- authority 文档通道：
  - `authority.document`
  - `authority.documentDiff`
- 运行反馈通道：
  - `runtimeFeedback`

文档中必须明确：

- 前者关注正式文档同步
- 后者关注运行态投影
- 二者会在 editor 同时出现，但不是同一回事

### 5.4 bundle 协议不能反向污染长期模型

当前 bundle 系统已经足够实用，但长期仍需坚持：

- `NodeModule`、`registerWidget(entry)`、`installNodeModule(...)` 是模型层与宿主层的正式接口
- `LeaferGraphEditorBundleBridge.registerBundle(...)` 是 editor 接入层协议
- 不能把 IIFE bridge 当成主包 API 的替代物

## 6. 接下来必须遵守的原则

### 6.1 以后端 authority 为默认事实真源

- 前端长期持有交互态、渲染态和未确认操作态
- 正式文档和正式操作模型默认以后端 authority 为准
- 本地 loopback 和浏览器 demo service 继续保留，但定位是验证链和离线回退链

### 6.2 继续保持 `packages/node`、`packages/leafergraph`、`packages/editor` 三层结构

- `packages/node` 不承担 Leafer 宿主职责
- `packages/leafergraph` 不承担 editor 页面壳层职责
- `packages/editor` 不复制主包内部图运行时

### 6.3 OpenRPC 真源只认 `openrpc/`

- 文档、生成器、测试和模板都应围绕 `openrpc/` 工作
- `LEAFERGRAPH_OPENRPC_ROOT` 是唯一统一覆盖入口
- 不再把旧模板共享 OpenRPC 目录写成现行路径

### 6.4 文档必须显式区分“当前实现”和“未来计划”

- 当前实现优先写
- 未来路线后写
- 历史方案只保留在明确标记区域

## 7. 分阶段计划

### 阶段 A：继续收敛 authority 与文档同步主线

目标：

- 让 authority 文档同步、diff、runtime feedback 和 bundle sync 的说明、测试与接线继续收口

需要完成：

- 持续补 remote authority 回归
- 保持 OpenRPC 生成代码与共享真源一致
- 继续降低 editor 对 transport 细节的散落依赖

完成标准：

- authority 文档通道和 runtime feedback 通道在文档、测试和 UI 中保持一致口径

### 阶段 B：继续收敛运行时与适配层

目标：

- 保持主包 runtime-agnostic，同时继续为外部应用接入做适配层准备

需要完成：

- 继续沉淀 `CapabilityProfile`
- 继续明确 `RuntimeAdapter` 与未来 `AppAdapter` 的边界
- 不把 Dora 或其它应用协议直接抬进核心层

完成标准：

- 新 runtime 或新应用优先通过 adapter 接入，而不是改写主模型

### 阶段 C：继续沉淀外部作者与分发路径

目标：

- 把模板、bundle、ESM 包、版本兼容和错误提示进一步稳定下来

需要完成：

- 继续整理 bundle manifest 约束
- 继续校准模板 README 与 editor loader 文档
- 持续让外部作者路径围绕正式模型而不是历史 demo 习惯

完成标准：

- 外部作者能区分正式包接入和 editor bundle 接入两条路径

## 8. 当前优先级建议

当前优先级从高到低建议为：

1. authority / OpenRPC / runtime feedback 文档与测试继续收口
2. 主包与 editor 边界继续守住
3. bundle/template 路径和外部作者说明继续收敛
4. 再考虑更高层的产品化体验补充
