## 1. 计划目标

这份计划书不再讨论“节点系统要不要做”，而是回答下面三个更具体的问题：

1. 现在节点系统已经做到哪一步
2. 节点系统接下来最应该先补哪些能力
3. 哪些事情应该算节点阶段目标，哪些事情应该留到图模型、编辑器或执行层再做

一句话总结：

- **当前节点系统已经完成“可定义、可注册、可实例化、可挂宿主、可走插件接入”的第一阶段，下一步要从 demo 级宿主接入推进到真正可编辑、可运行、可扩展的节点运行时。**

---

## 2. 当前基线

结合现有代码和现有文档，当前节点系统已经具备以下基础：

### 2.1 `@leafergraph/node` 已具备的能力

- 类型层：
  - `NodeDefinition`
  - `WidgetDefinition`
  - `NodeRuntimeState`
  - `NodeModule`
  - `NodeModuleScope`
- 注册层：
  - `NodeRegistry`
  - `WidgetRegistry`
- 实例层：
  - `createNodeState()`
  - `configureNode()`
  - `serializeNode()`
  - `createNodeApi()`
- 模块层：
  - `installNodeModule()`
  - `resolveNodeModule()`
  - `resolveScopedNodeType()`
- 生命周期层：
  - `onCreate`
  - `onConfigure`
  - `onSerialize`
  - `onExecute`
  - 以及输入输出、属性、连接相关回调

### 2.2 主包 `leafergraph` 已具备的能力

- 主包已持有唯一 `NodeRegistry`
- 主包已支持：
  - `use(plugin)`
  - `installModule(module)`
  - `registerNode()`
  - `registerWidget()`
  - `registerWidgetRenderer()`
  - `getWidgetRenderer()`
- 主包已暴露：
  - `LeaferUI`
  - `LeaferGraphNodePlugin`
  - `LeaferGraphWidgetRenderer`
  - 最小 `mount / update / destroy` renderer 协议
- demo 节点已改为使用真实 `NodeRuntimeState`
- Widget 已不再只是硬编码展示，而是走 renderer 实例调度链路

### 2.3 当前仍然是“半成品”的部分

虽然上述基础已经齐了，但当前实现仍主要停留在：

- demo 场景驱动
- 节点视图由主包单文件直接绘制
- 图模型尚未独立成完整 `GraphModel`
- 连线只有 demo 级数据来源
- 节点还没有进入真正的“可编辑节点运行时”

也就是说，当前状态更准确的描述应该是：

- **节点 SDK 已成立**
- **插件协议已成立**
- **宿主接线已成立**
- **但节点系统还没有形成完整产品级闭环**

---

## 3. 当前主要问题

结合现有实现，下一阶段最核心的问题有六个。

### 3.1 节点数据来源仍偏 demo

现在主包虽然已经能吃 `NodeRuntimeState`，但初始化入口仍然明显偏向 demo 数据：

- `LeaferGraphNodeData`
- 默认节点数组
- 顺序自动连线

问题在于：

- 节点创建入口还不是统一的正式 API
- 节点删除、复制、移动、重建没有统一命令面
- 图内节点集合还没有独立模型层

### 3.2 节点壳与节点宿主仍耦合在主包大文件中

当前节点卡片壳、槽位文本、Widget 区、状态灯、分类胶囊都直接写在主包渲染逻辑里。

这意味着：

- 节点壳还不是独立的 host / renderer 抽象
- 后续如果做多节点样式、多主题、多节点变体，会很快膨胀
- widget 区虽然已经支持 renderer，但节点整体壳层还没有组件化

### 3.3 Widget 只打通了渲染协议，还没打通交互协议

现在已经有：

- Mount
- Update
- Destroy

但还没有统一约定：

- widget 如何向宿主回写值
- widget 如何声明可编辑状态
- widget 如何参与命中测试、键盘输入、焦点和拖拽

这会限制后续真正可用的参数控件系统。

### 3.4 节点生命周期还没有接进图执行与连接系统

`NodeLifecycle` 现在已经存在，但还缺少真正的宿主调度时机：

- `onExecute` 何时被调
- `onConnectionsChange` 由谁驱动
- `getInputData / setOutputData` 如何进入图级运行时

这说明当前节点系统还没有和真正的图运行时闭环。

### 3.5 外部节点接入已能注册，但开发体验还不完整

现在外部包已经可以：

- 声明模块
- 声明插件
- 注册节点
- 注册 Widget
- 注册 Widget renderer

但还缺：

- 标准模板
- 最小示例工程
- 版本兼容约束
- 插件安装结果可视化
- 节点搜索、分组、创建菜单的正式消费面

### 3.6 节点系统和图系统边界还没有完全成型

当前文档上已经明确：

- 节点系统
- 图模型
- scene sync
- editor 壳层

应该分层。

但代码上当前真正落地的还主要是：

- 节点 SDK
- demo 宿主

所以接下来必须避免把“图命令、视口、选择、执行、搜索”继续直接塞进当前节点渲染文件。

---

## 4. 下一阶段的总体原则

接下来节点系统演进时，建议始终遵守以下原则。

### 4.1 先把“节点系统”做实，再把“编辑器体验”做厚

优先顺序应该是：

1. 节点实例与图模型关系
2. 节点宿主与渲染同步
3. widget 交互协议
4. 编辑器面板和视觉增强

而不是反过来先继续堆 demo UI。

### 4.2 节点 SDK 继续保持宿主无关

`@leafergraph/node` 应继续只关心：

- 定义
- 状态
- 注册
- 配置
- 序列化
- 模块安装

不要把 Leafer 图元、指针事件、DOM 面板状态塞进 SDK。

### 4.3 宿主只负责调度，不侵入节点 authoring

主包应该负责：

- 创建节点视图
- 调度 lifecycle
- 调度 widget renderer
- 管理节点与连线的场景同步

但不要把节点 authoring 写法重新绑死回主包内部私有格式。

### 4.4 节点壳要从“当前 demo 实现”升级为“可演进的 host”

下一阶段不能只是在 `index.ts` 里继续加分支。

需要有更明确的抽象，例如：

- `NodeShellLayout`
- `NodeShellRenderer`
- `NodeWidgetHost`
- `NodePortLayout`

至少要开始拆出方向，而不是继续堆单文件。

---

## 5. 分阶段计划

建议把当前节点工作拆成五个阶段，按顺序推进。

---

## 阶段 A：节点实例入口正式化

### 目标

把当前 demo 数据入口推进成真正的节点实例操作入口。

### 需要完成

- 为主包增加正式节点操作 API：
  - `createNode(...)`
  - `removeNode(...)`
  - `updateNode(...)`
  - `moveNode(...)`
- 为节点实例集合建立宿主级容器，而不是只靠 demo 数组
- 把“默认顺序自动连线”从初始化流程中拿掉，改成显式图数据

### 完成标准

- 主包不再依赖 demo 顺序来构建节点关系
- 可以通过正式 API 增删改节点
- 节点创建不必再依赖 `LeaferGraphNodeData` 的演示字段集合

### 这一阶段不做

- 完整执行引擎
- 复杂 widget 交互
- 节点模板市场

---

## 阶段 B：节点宿主抽象化

### 目标

把当前直接写在主包里的节点卡片绘制逻辑，抽成可维护的节点宿主。

### 需要完成

- 抽出节点壳层布局数据：
  - header
  - slots area
  - widget area
- 抽出节点壳层绘制函数或 host 类：
  - `createNodeShell(...)`
  - `updateNodeShell(...)`
- 抽出节点端口布局计算：
  - 输入端口位置
  - 输出端口位置
  - 首端口主锚点位置

### 完成标准

- 节点卡片壳不再散落在主包单个大函数中
- 节点壳层和 widget 渲染层边界明确
- 后续可以替换主题或视觉变体，而不需要重写整个节点创建函数

### 这一阶段不做

- 多主题系统
- 动画系统
- 节点折叠全部状态逻辑

---

## 阶段 C：Widget 交互协议落地

### 目标

让 Widget 不止能显示和更新，还能真正参与交互。

### 需要完成

- 为 renderer 上下文补最小回写接口，例如：
  - `setValue(...)`
  - `requestRender(...)`
  - `emitAction(...)`
- 约定 widget 与宿主的值同步流程：
  - 用户交互
  - 宿主写回 `widget.value`
  - 宿主触发实例 `update()`
- 明确 widget 销毁时的解绑责任

### 完成标准

- slider / toggle / combo 至少有一种能真正双向工作
- widget 改值不需要外部手工调用底层方法修补
- widget 生命周期和宿主状态同步关系明确

### 这一阶段不做

- 完整焦点管理系统
- 富文本编辑 widget
- HTML overlay widget

---

## 阶段 D：节点接入图模型与连线系统

### 目标

把节点系统从“独立卡片 + demo 连线”推进到“真正属于图模型的一部分”。

### 需要完成

- 定义图级数据结构：
  - 节点集合
  - 连线集合
  - 可选分组集合
- 让节点移动、删除、重建能驱动连线同步
- 让连接变化能真正进入：
  - `onConnectionsChange`
  - 输入输出缓存更新

### 完成标准

- 连线不再只是两个 demo 节点之间的展示对象
- 节点变化和连线变化走统一数据源
- 生命周期中的连接回调开始有真实调用场景

### 这一阶段不做

- 完整命令栈
- 历史回放
- 远程协作

---

## 阶段 E：节点执行与外部生态完善

### 目标

让节点从“可编辑结构”进入“可执行结构”，并把外部节点生态补到可用。

### 需要完成

- 建立最小执行调度：
  - 输入值读取
  - 输出值写回
  - 节点执行时机
- 为外部节点包提供：
  - 最小模板
  - 带 widget 模板
  - 带执行逻辑模板
- 为主包补：
  - 搜索面
  - 菜单分组面
  - 插件安装结果可观察能力

### 完成标准

- 至少存在一类真正可执行的外部节点
- 节点执行结果能反映到 widget 或状态表现上
- 外部作者能按模板独立开发节点包并接入主包

### 这一阶段不做

- 插件市场
- 权限系统
- 完整版本协商机制

---

## 6. 优先级建议

如果只按“现在最值得做什么”排序，我建议是：

1. 阶段 A：节点实例入口正式化
2. 阶段 B：节点宿主抽象化
3. 阶段 C：Widget 交互协议落地
4. 阶段 D：节点接入图模型与连线系统
5. 阶段 E：节点执行与外部生态完善

原因很简单：

- 现在最缺的不是再补一个 renderer，而是把节点从 demo 数据里解放出来
- 当前最危险的技术债不是 API 少，而是节点壳继续堆在主包单文件中
- 如果不先把实例入口和宿主边界做干净，后面执行系统和外部生态都会被反向拖慢

---

## 7. 风险与控制

### 风险 1：继续在 demo 层叠功能

后果：

- 节点系统会越来越像临时页面逻辑
- 后续再抽象时成本更高

控制方式：

- 新功能优先以宿主 API 或节点抽象落地
- 少往默认 demo 数据结构里继续加私有字段

### 风险 2：SDK 过早膨胀

后果：

- `@leafergraph/node` 会重新和宿主耦合
- 外部节点作者反而更难理解

控制方式：

- SDK 只保留宿主无关能力
- Leafer 相关能力继续留在主包或宿主上下文

### 风险 3：Widget 系统过早做成“大一统框架”

后果：

- 第一阶段实现成本过高
- 大量接口会先设计后废弃

控制方式：

- 保持最小协议
- 先把 `mount / update / destroy + setValue` 路径做顺

---

## 8. 阶段性验收标准

当下面这些条件同时满足时，可以认为“当前节点计划书”的核心目标达成：

- 节点创建、删除、移动有正式 API，而不是只靠 demo 初始化
- 节点壳和 widget 渲染从主包大文件中开始分层
- Widget 至少存在一类可交互、可回写、可销毁的正式实现
- 节点与连线共享统一数据源
- 外部节点包能通过模板方式接入，而不是只靠手写示例

---

## 9. 当前结论

当前节点系统已经跨过“概念设计”和“纯文档阶段”，进入了一个很关键的中间点：

- SDK 已成立
- 插件协议已成立
- 宿主接线已成立
- 但真正的节点运行时、节点宿主分层、widget 交互、图模型闭环还没有完成

因此，接下来最重要的不是继续补零散功能，而是按顺序把下面这条主线做实：

**节点实例入口正式化 -> 节点宿主抽象化 -> Widget 交互协议 -> 图模型闭环 -> 执行与生态完善**

这条主线一旦站稳，后面的编辑器功能、搜索、菜单、执行反馈、模板和生态都会更顺。
