## 1. 这份文档解决什么问题

`progress ring` 是节点最外层 rounded-rect 外环，用来表达“这个节点当前是不是长任务，以及它的粗粒度进度”。

这份文档重点回答：

1. 什么时候该显示进度环
2. 节点作者该怎么声明和投影进度
3. `Delay`、`Crawler` 这两类常见长任务该怎么写
4. 哪些写法看起来能跑，但会导致进度环不显示、闪一下就没了，或者状态和业务 UI 打架

## 2. 两种进度环模式

### 2.1 determinate

当执行态里提供了明确进度值时，显示 determinate 进度环：

```ts
state: {
  status: "running",
  progress: 0.42
}
```

语义：

- `progress` 范围固定是 `0..1`
- 只在 `status === "running"` 时有效
- 适合真实可计算百分比的任务

典型场景：

- `Delay` 这类 timeout 已知总时长
- 文件上传、下载、转码
- 分页抓取、批量处理、流水线阶段总数明确

### 2.2 indeterminate

当节点处于 `running`，但没有精确 `progress` 时，显示 indeterminate 运行环。

前提：

```ts
shell: {
  longTask: true
}
```

语义：

- 表示“这个节点正在跑一个长任务”
- 不表示业务百分比
- 只在没有 determinate progress 时兜底显示

典型场景：

- `Crawler` 这类 fetch 完成时间不确定
- 等待远端接口、等待模型响应、等待系统回调
- 有长任务感知，但没有可靠百分比

## 3. 作者侧最小接入方式

如果你的节点只是“会长时间 running，但没有精确百分比”，最小接入是：

```ts
static meta = {
  type: "demo/long-task",
  title: "Long Task",
  shell: {
    longTask: true
  }
};
```

这样当节点进入 `running` 后，shell 会显示 indeterminate 进度环。

如果你的节点还能提供真实进度，就继续投影：

```ts
state: {
  status: "running",
  progress: 0.5
}
```

此时 determinate 会优先于 indeterminate。

## 4. 推荐写法

### 4.1 已知总时长任务

优先使用图定时器并开启 `trackProgress: true`。

参考 [`event_timing.ts`](../example/authoring-basic-nodes/src/developer/nodes/event_timing.ts) 里的 `DelayEventNode`：

```ts
runtimePayload.registerGraphTimer({
  nodeId: ctx.node.id,
  runId: execution.runId,
  source: execution.source,
  startedAt: execution.startedAt,
  intervalMs: delayMs,
  immediate: false,
  timerId: ticketId,
  mode: "timeout",
  trackProgress: true
});
```

这类写法的优点：

- 进度环天然是 determinate
- 宿主统一投影 `0..1`
- 作者不用自己做百分比时钟

### 4.2 未知时长异步任务

声明 `shell.longTask: true`，进入运行后保持节点处于 `running`，直到任务真正结束。

参考 [`crawler.ts`](../example/web-crawler-nodes/src/developer/nodes/crawler.ts)：

- `Crawler` 声明了 `shell.longTask: true`
- 任务开始后先更新 widget 状态为 `RUNNING`
- 图执行路径下通过 poll timer + `preserveRunningExecutionState(...)` 保持宿主看到节点仍处于 `running`
- 任务完成后再输出结果并进入 `done / success`

适合：

- `fetch`
- websocket 首包等待
- 后台 job 轮询
- 任何“只知道在跑，但不知道跑到多少”的任务

## 5. 宿主与壳体当前行为

### 5.1 进度环显示条件

shell 当前只看执行态，不看 widget 内容，不看 `properties`，也不看 `node.data` 里的业务字段。

规则是：

1. `status === "running"` 且 `progress` 是数字：显示 determinate ring
2. `status === "running"`、`shell.longTask === true`、但没有 `progress`：显示 indeterminate ring
3. `idle / success / error`：清空进度环

### 5.2 固定起始点

当前 indeterminate 进度环已经调整为：

- 每次进入新的 `running` 周期时，从固定起始点开始
- 同一次运行内按 elapsed time 连续推进
- 退出 `running` 后重置

也就是说：

- 同一轮任务里，环会连续转动
- 下一轮新任务开始时，会重新从固定起点起跑

### 5.3 Reduced Motion

当宿主启用 `prefers-reduced-motion: reduce` 并选择遵循时：

- 不播放 sweep 动画
- 改为静态 partial ring
- 不会把“正在运行”这个信号隐藏掉

## 6. 常见坑

### 6.1 只改 widget 文案，不改执行态

错误示例：

```ts
updateStatus(ctx, "RUNNING");
```

但没有让节点执行态进入 `running`。

结果：

- widget 看起来在跑
- shell 不会显示进度环

原因：

- shell 不读取 widget 文案
- shell 只看 execution state

### 6.2 节点会长期 running，但没声明 `shell.longTask`

结果：

- 节点状态灯会是 running
- 但外层不会出现 indeterminate 进度环

适合修复：

```ts
shell: {
  longTask: true
}
```

### 6.3 有长任务，但没有真正保持 `running`

常见表现：

- 触发后瞬间 success
- 外环只闪一下，甚至完全看不到

常见原因：

- 任务开始后马上返回 success
- 异步任务没有被宿主感知为仍在运行
- 图定时器注册了，但当前执行结束时没有保留 running override

对于图执行里的“任务尚未完成、但当前步已经结束”的场景，通常需要：

```ts
preserveRunningExecutionState(ctx);
```

### 6.4 把业务百分比塞进 widget，期望 shell 自动识别

错误思路：

- widget 上显示了 `42%`
- 以为壳体会自己读出来

不会生效。

v1 规则固定是：

- shell 只认 `execution.state.progress`

### 6.5 `progress` 不是 `0..1`

错误示例：

```ts
progress: 42
progress: 80
```

正确做法：

```ts
progress: 0.42
progress: 0.8
```

### 6.6 进入 `success / error / idle` 后还残留 progress 语义

规范要求：

- 非 `running` 状态下，进度环应该被清空
- 不要继续依赖旧 progress 让 UI 维持上一次百分比

### 6.7 同一次任务里反复重建“起始时刻”

如果作者在业务层自己维护动画相位，不要在每次刷新时都重置起点。

正确语义应该是：

- 新任务开始：建立起点
- 同一任务刷新：沿用起点
- 任务结束：清理起点

## 7. 什么时候该用 determinate，什么时候该用 indeterminate

优先级建议：

1. 能拿到真实百分比，就用 determinate
2. 只能确认“仍在跑”，就用 indeterminate
3. 根本不是长任务，就不要声明 `shell.longTask`

简单判断法：

- 问题是“已经完成多少”：用 determinate
- 问题只是“是不是还没结束”：用 indeterminate

## 8. `Delay` 与 `Crawler` 的对照

### 8.1 `Delay`

特点：

- 总时长已知
- timeout 天然可以算出百分比

推荐：

- `shell.longTask: true`
- 图定时器 `trackProgress: true`

收益：

- determinate ring
- 进度准确
- shell 与 runtime 语义一致

### 8.2 `Crawler`

特点：

- 请求是否很快完成不可预知
- 大多数时候只能确认“正在请求中”

推荐：

- `shell.longTask: true`
- 进入运行后保持 execution state 为 `running`
- 需要时用 poll timer 把长任务维持到真正完成
- 业务阶段名、URL、返回大小、错误提示放 widget

收益：

- shell 负责全局稳定信号
- widget 负责节点内部细节

## 9. 给节点作者的检查清单

- 这个节点真的会长时间停留在 `running` 吗
- 如果会，是否声明了 `shell.longTask: true`
- 如果能提供真实百分比，是否投影到了 `execution.state.progress`
- `progress` 是否使用 `0..1`
- 节点进入 `success / error / idle` 后是否会清空进度态
- widget 里是否只放业务细节，而不是拿它代替 shell 信号
- 如果是异步图任务，当前执行结束后是否正确保留了 `running`
- 是否考虑了 reduced motion 下的可见性

## 10. 一句话原则

`progress ring` 只表达“这个节点是不是长任务，以及它大概完成了多少”；业务阶段、日志、URL、错误提示、百分比标签都继续放 widget，不要让 shell 猜。
