# LeaferGraph 范围划定与设计选项

## 文档信息

- 当前状态：现状优先，保留设计判断
- 最近校对：2026-03-25
- 适用范围：仓库级范围、包级职责、长期演进方向

## 1. 当前问题定义

LeaferGraph 当前不是在回答“能不能做一个节点 demo”，而是在回答更具体的问题：

1. 如何把节点图模型、Leafer 渲染宿主和 editor 壳层拆成可维护的三层。
2. 如何让 authority、OpenRPC、bundle/template、runtime feedback 同时成立，而不互相污染。
3. 如何在保留本地验证链的同时，默认以后端 authority 作为长期事实真源。

这意味着这份文档的范围必须围绕“当前真实仓库”来写，而不是围绕一套已经过时的理想目录。

## 2. 当前范围划定

### 2.1 `packages/node` 的范围

当前 `packages/node` 负责：

- `NodeDefinition`
- `NodeModule`
- `NodeRegistry`
- `GraphDocument`
- `CapabilityProfile`
- 节点序列化、配置与模块安装

当前不应放进 `packages/node` 的内容：

- Leafer 场景对象
- editor UI
- authority transport
- browser bundle 装载逻辑

### 2.2 `packages/authoring` 的范围

当前 `packages/authoring` 负责：

- 面向节点 / widget 作者的干净前端 authoring SDK
- `BaseNode` / `BaseWidget` 作者类
- `defineAuthoringNode(...)`
- `createAuthoringModule(...)`
- `defineAuthoringWidget(...)`
- `createAuthoringPlugin(...)`
- 把作者代码收口为 `NodeDefinition`、`NodeModule`、`LeaferGraphWidgetEntry`、`LeaferGraphNodePlugin`

当前不应放进 `packages/authoring` 的内容：

- `examples/editor` 依赖
- browser bundle helper
- loader manifest / bundle bridge
- authority session / transport
- 为 editor 或历史链路服务的反向兼容适配

### 2.3 `packages/leafergraph` 的范围

当前 `packages/leafergraph` 负责：

- Leafer-first 图运行时宿主
- 图文档装配与整图替换
- `GraphOperation` 应用
- `GraphDocumentDiff` 投影
- 节点壳、连线、交互、菜单基础设施
- 运行反馈投影

当前不应放进主包的内容：

- workspace 页面布局
- authority transport 实现
- editor 专属 bundle 协议
- 外部应用专属产品语义

### 2.4 `examples/editor` 的范围

当前 `examples/editor` 负责：

- Preact 壳层
- `GraphViewport`
- 命令总线与 history
- 右键菜单语义与接线
- bundle loader 与浏览器持久化
- authority session 与 transport 适配

当前不应把 editor 写成：

- 图模型定义层
- 主包运行时复制品
- OpenRPC 真源目录

## 3. 当前真实设计选项

### 3.1 已经淘汰的方向

以下方向当前不再推荐当作现行方案：

- 单体式“一个包包办所有模型、渲染、编辑器和 authority”
- 继续围绕旧 demo 数据结构扩核心 API
- 把 bundle 协议直接抬成主包公共接口
- 针对单一外部应用把协议硬编码进核心层

### 3.2 当前实际采用的方向

当前仓库实际采用的是“核心三层 + 作者层 + 协议真源外置”的方案：

| 层级 | 当前落点 | 说明 |
| :--- | :--- | :--- |
| 节点模型层 | `packages/node` | 统一节点与图文档模型 |
| 作者层 | `packages/authoring` | 统一节点 / widget 作者体验，并输出正式注册产物 |
| 图运行时层 | `packages/leafergraph` | Leafer 渲染宿主、交互与正式图 API |
| 编辑器壳层 | `examples/editor` | 命令、viewport、authority、bundle、UI |
| 协议真源 | `openrpc/` | authority OpenRPC 文档、schema 与 conformance |

这是当前最接近仓库现实的设计方案。

## 4. 为什么当前方案更合适

### 4.1 它允许 authority-first，同时保留本地验证链

当前这套结构允许：

- 后端 authority 维护正式 `GraphDocument`
- editor 通过 session 消费 `authority.document` / `authority.documentDiff`
- 本地 loopback 和本地 demo service 继续存在

但不会逼我们把浏览器内 demo service 误写成长期事实真源。

### 4.2 它允许 bundle/template 保持在 editor 接入层

当前 bundle/template 已经是重要能力，但它们被放在：

- `examples/editor/src/loader/`
- `templates/misc/browser-node-widget-plugin-template`

而不是反向污染主包。这一点是当前方案的关键收益。

### 4.3 它允许 OpenRPC 真源脱离单个模板

现在 OpenRPC 已经从旧模板共享目录迁到仓库根 `openrpc/`，这带来两个直接好处：

- editor 生成器和 Python 模板围绕同一真源工作
- 文档、conformance、生成器和测试不再依赖模板内部路径

## 5. 当前边界上的固定判断

### 5.1 authority 方向

长期默认判断：

- 正式文档与正式操作模型以后端 authority 为主
- 前端长期持有交互态、渲染态和未确认操作态
- 本地执行器与本地 authority demo 保留，但定位是验证链和离线回退

### 5.2 OpenRPC 方向

当前固定判断：

- 真源目录是 `openrpc/`
- 全仓统一环境变量是 `LEAFERGRAPH_OPENRPC_ROOT`
- editor 协议消费已经开始收口到 `examples/editor/src/session/authority_openrpc/`

### 5.3 demo / bundle 方向

当前固定判断：

- `demo / node / widget` 仍然是 editor 装载槽位的真实名字
- 它们不是长期核心领域模型的唯一语言
- 文档可以使用“蓝图 / 节点 / 组件”帮助解释，但必须保留代码命名映射

## 6. 推荐路线

### 6.1 当前近期路线

当前推荐继续沿着这几条线推进：

1. authority/OpenRPC/session/runtime feedback 文档和测试继续收口
2. 主包与 editor 边界继续守住
3. bundle/template 与外部作者路径继续沉淀

### 6.2 当前中期路线

中期仍然建议持续推进：

- `CapabilityProfile`
- `RuntimeAdapter`
- 未来 `AppAdapter`
- 多应用接入而非单应用硬编码

### 6.3 当前不建议优先做的事

当前不建议把主要精力重新放回：

- 再补一套旧 demo 风格入口
- 发明新的不存在目录结构
- 把 editor 专属便利协议挪进主包

## 7. 对文档与代码的约束

从这份文档出发，后续任何新文档或设计说明都应默认满足：

- 当前目录和当前入口必须可在仓库中找到
- 当前实现和未来方案必须明确分段
- 旧模板共享 OpenRPC 目录只能出现在历史说明里
- 不再把旧 GraphViewport 入口、旧 `context_menu.ts` 单文件入口这类历史路径写成现状

## 8. 当前结论

当前最合理的范围表达不是“做一个大而全节点平台”，而是：

- 用 `packages/node` 承接模型
- 用 `packages/authoring` 承接干净作者层与正式注册产物组装
- 用 `packages/leafergraph` 承接 Leafer-first 运行时
- 用 `examples/editor` 承接工作区壳层
- 用 `openrpc/` 承接 authority 协议真源

这套范围划定已经落地到当前仓库，不需要再回退到旧 demo 结构或重新发明一套不存在的目录。
