# LeaferGraph 范围划定与设计方案

## 文档信息

- 日期：`2026-03-11`
- 适用对象：`leafergraph` 新实验工程
- 当前前提：
  - 目标是做一套干净的 Leafer-first 节点图
  - 不承担旧版本兼容和现网稳定性负担
  - 需要覆盖 LiteGraph 现有大部分面向用户的能力，但暂不做 `WebGL`
  - 需要考虑“外部独立节点”的接入与模板化

## First Principles Analysis: LeaferGraph 范围与方案

### 1. Problem Essence

**Core problem:**  
要定义的不是“把 LiteGraph 代码搬多少过来”，而是“新 LeaferGraph 要提供哪些用户能力，以及这些能力由什么层来承担”。

**Success criteria:**  

1. 新工程能覆盖除 `WebGL` 外，LiteGraph 大部分用户能直接感知到的功能与设置
2. 内核保持干净，不被旧兼容层和历史 API 绑住
3. 编辑器功能和节点运行时边界清楚
4. 外部独立节点能以模板方式接入，而不是只能复制仓库内部代码
5. 第一版就能形成可扩展的目录结构，不必后面再推翻

### 2. Assumptions Challenged

| Assumption | Challenge | Verdict |
|------------|-----------|---------|
| “功能范围 = 把 LiteGraph 所有东西都做一遍” | 用户看到的是能力集合，不是历史实现；同一能力可以换更干净的实现 | Modify |
| “不做兼容，就可以先不设计扩展” | 没有扩展设计，后面外部节点就会反向污染核心库 | Discard |
| “先把 editor 做完，再考虑库” | 节点图的长期可维护性更多取决于库和 editor 的边界 | Discard |
| “外部节点模板化只是脚手架问题” | 它本质上是节点契约、注册机制、资源组织方式的问题 | Discard |
| “范围先定得越全越好” | 过度铺开会让第一阶段失去验证重点，应该先定能力边界，再定阶段优先级 | Modify |

### 3. Ground Truths

- 节点图产品的核心不是某个渲染 API，而是：图模型、编辑交互、执行调度、节点扩展机制。
- 用户真正关心的是“能不能编辑、能不能运行、能不能扩展、能不能配置”，而不是内部是否叫 LiteGraph。
- 如果外部独立节点是目标能力，那么节点定义必须从第一版起就是“包外可写”的，而不是仓库内专用写法。
- `WebGL` 不做，意味着第一阶段可以明确聚焦在 retained-mode 2D 图编辑器问题本身。
- 当前工程已经有清晰起点：
  - 库：`packages/leafergraph`
  - 编辑器：`packages/editor`
  - 编辑器由 Preact 控制

### 4. Reasoning Chain

Ground Truth  
-> 先按“能力层”划范围，而不是按旧文件划范围  
-> 再根据“内核纯度 / 覆盖率 / 扩展能力”来出方案  
-> 最终选择一个既能覆盖 LiteGraph 用户能力、又能支持外部节点模板化的结构

### 5. Conclusion

**Recommended approach:**  
采用“核心库 + 编辑器壳 + 节点模板/SDK”思路来规划，而不是单体式 editor-first。

**Key insight:**  
这次真正要定的范围有三层：

- 运行时能力范围
- 编辑器能力范围
- 外部节点生态范围

**Trade-offs acknowledged:**  
这样做前期抽象会比“直接往 editor 里堆功能”稍多，但它能避免后面节点生态反向绑死核心库。

## 一、建议先这样划范围

先不要把范围写成“实现哪些源码文件”，而是写成下面三层。

### A. 运行时内核范围

这一层属于 `packages/leafergraph`，职责是“没有 editor UI 也能跑图”。

建议纳入范围：

- graph / node / link / group 基础数据结构
- 节点注册与分类系统
- slot / widget / property 基础契约
- scene sync 与 Leafer 宿主
- 视口坐标体系与命中测试基础能力
- 执行模型：`play / step / stop`
- 最小序列化 / 反序列化
- 运行时设置项
  - 画布显示
  - 连线表现
  - 节点外观
  - 执行可视反馈

明确不纳入第一阶段：

- `WebGL`
- 旧版 canvas 兼容 API
- global bridge / CJS bridge
- 历史序列化兼容修补

### B. 编辑器功能范围

这一层属于 `packages/editor`，职责是“把用户可见的编辑体验组织起来”。

建议覆盖 LiteGraph 面向用户的大部分能力，但按模块组织，而不是按旧实现照搬：

- 基础编辑
  - 平移、缩放、框选、多选、拖拽
  - 拉线、断线、重连
  - 节点创建、删除、复制、粘贴
- 画布与节点交互
  - 右键菜单
  - 搜索框
  - 属性面板
  - widgets 编辑
  - 节点折叠、固定、分组
- 执行与调试
  - `play / stop / step`
  - 执行高亮
  - 连线活动态反馈
  - 基础运行状态栏
- 设置系统
  - editor 级设置
  - graph 级设置
  - 渲染表现设置
  - 交互行为设置
- 快捷键系统
  - 视口快捷键
    - `Space + Drag` 平移
    - 鼠标滚轮缩放
  - 选择快捷键
    - `Ctrl` / `Shift + Click` 多选
    - `Ctrl + A` 全选
    - `Esc` 取消当前选择或关闭浮层
  - 编辑快捷键
    - `Ctrl + C` 复制
    - `Ctrl + V` 粘贴
    - `Delete` / `Backspace` 删除
    - `Alt + Drag` 快速克隆节点
  - 运行快捷键
    - 为 `Play / Stop / Step` 预留快捷键入口
  - 菜单与搜索快捷键
    - 为搜索框、节点创建、面板开关预留键位映射

这里的关键是：

- “覆盖 LiteGraph 面向用户的大部分功能”是**能力目标**
- 不是要求第一版就逐项像素级还原

### C. 外部独立节点生态范围

这一层决定后面节点图是不是能“长出生态”，不应该等到最后才补。

建议直接纳入范围：

- 外部节点包的标准入口
- 节点元数据声明格式
- 节点注册 API
- 节点分类、搜索关键词、展示文案契约
- 外部节点模板
  - 最小节点模板
  - 带 widget 的模板
  - 带执行逻辑的模板
- 本地开发联调方式
- 编辑器侧如何发现和加载外部节点

如果这一层不先设计，后面很容易出现：

- 节点只能写在仓库内部
- 每个节点作者都用不同格式
- editor 和 library 对节点的理解不一致

## 二、建议的功能清单表达方式

你提到“除了 `WebGL` 外基本所有 LiteGraph 的面向用户功能和设置”，这个方向我认同，但建议写成下面这种“能力清单”，而不是散点需求。

### 1. 图编辑能力

- 创建图、加载图、保存图
- 节点增删改查
- 连线增删改查
- 分组与层级辅助
- 复制、粘贴、删除、框选、多选

### 2. 交互能力

- 平移、缩放、定位
- 端口命中测试
- 拉线预览与连接反馈
- 右键菜单与快捷入口
- 搜索创建节点

### 3. 节点展示能力

- 标题、颜色、尺寸、形状
- 插槽展示
- widgets 展示
- 状态与执行反馈
- 折叠态与选中态

### 4. 执行能力

- `play / stop / step`
- 节点执行顺序
- action / event 传播
- 执行高亮与链路反馈

### 5. 配置能力

- editor 设置
- graph 设置
- 渲染表现设置
- 交互行为设置
- 节点默认配置

### 6. 外部节点能力

- 安装 / 引入外部节点包
- 注册节点分类与元信息
- 模板化生成节点
- 外部资源与样式约定
- 开发环境联调

## 三、三个设计方案

下面这三个方案不是“技术选型不同”，而是“边界和复杂度不同”。

### 方案 A：单体式方案

结构思路：

- `packages/leafergraph` 既放运行时，又逐步吸收 editor 逻辑
- `packages/editor` 更像 demo / 壳
- 外部节点先通过简单注册函数接入

优点：

- 上手最快
- 前期文件最少
- 适合快速把用户功能铺出来

缺点：

- editor 和 library 边界很容易混掉
- 后面做外部节点模板化时，契约容易漂移
- 长到中后期会再次出现“大核心”问题

适合：

- 非常短周期验证
- 只想先把一批功能做出来

不适合：

- 你现在这种“想维护一套干净代码”的目标

### 方案 B：双层方案

结构思路：

- `packages/leafergraph`
  - 图模型
  - 节点注册
  - scene sync
  - 执行内核
  - 运行时设置
- `packages/editor`
  - Preact editor shell
  - 工具栏、菜单、面板、搜索、状态管理
  - 编辑器设置
- 外部节点
  - 先用清晰的节点契约
  - 再配一个模板目录或脚手架

优点：

- 结构最平衡
- 能覆盖大部分 LiteGraph 用户功能
- 又不会让 editor 反向绑死核心库
- 外部节点模板化比较自然

缺点：

- 需要从一开始就明确哪些逻辑归库、哪些逻辑归 editor
- 前期抽象成本比方案 A 稍高

适合：

- 当前这个项目阶段

我的建议：

- **优先选这个**

### 方案 C：平台化方案

结构思路：

- `packages/leafergraph`：纯核心运行时
- `packages/editor`：官方编辑器
- `packages/node-sdk`：节点契约与 authoring helper
- `packages/node-templates`：节点模板
- `packages/node-presets`：官方节点集

优点：

- 生态边界最清楚
- 外部节点接入最规范
- 后续做社区节点、官方节点包都更顺

缺点：

- 现在就上这套，前期包会偏多
- 在功能还没收敛前，组织成本略高

适合：

- 当你已经确定要把外部节点生态作为核心目标
- 或者第二阶段从方案 B 平滑升级过来

---

## 四、推荐路线

我建议采用：

- **范围按 A/B/C 三层划定**
- **结构先用方案 B**
- **外部节点模板化按方案 C 的方向预留接口**

也就是：

1. 现在不做单体式大核心
2. 现在也不急着拆出太多包
3. 先把 `library` 和 `editor` 两层边界做对
4. 外部节点契约从第一版开始就按“包外可写”设计

---

## 五、建议的目录目标

如果按方案 B 走，建议目标结构可以先瞄准这个方向：

```text
leafergraph/
  packages/
    leafergraph/
      src/
        model/
        runtime/
        render/
        interaction/
        registry/
        settings/
        serialization/
        index.ts
    editor/
      src/
        app/
        panels/
        toolbar/
        menus/
        search/
        state/
        adapters/
        components/
        main.tsx
  templates/
    node-basic/
    node-widget/
    node-executable/
  docs/
    范围与设计选项.md
```

这里的重点不是目录名字本身，而是：

- 核心库不要长出 editor UI
- editor 不要承担图执行内核
- 节点模板不要藏在 editor 里

## 六、我建议你接下来先定的三件事

### 1. 明确“第一阶段必须覆盖”的 LiteGraph 用户能力

建议先列成能力矩阵，而不是需求散点。

### 2. 明确设置系统的分层

最少区分：

- runtime 设置
- editor 设置
- node 元数据默认设置

### 3. 明确外部节点契约

至少先定：

- 节点入口长什么样
- 节点元数据长什么样
- 节点如何注册
- editor 如何发现并展示它

---

## 最终建议

一句话结论：

- 你的范围判断方向是对的：`WebGL` 先排除，目标覆盖 LiteGraph 大部分面向用户的功能和设置，并且把外部独立节点模板化纳入核心范围
- 结构上不要走单体式，优先走“核心库 + Preact editor + 外部节点模板预留”的双层方案
- 第一版不需要一次性把所有功能做完，但必须先把范围框架和扩展契约定对

下一步最值得补的是一份“能力矩阵”，把 LiteGraph 用户功能拆成：

- 必做
- 应做
- 暂缓

这样后面就能直接转成开发阶段计划。
