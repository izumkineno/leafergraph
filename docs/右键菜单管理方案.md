# 右键菜单管理方案

## 文档信息

- 当前状态：现状说明
- 最近校对：2026-03-23
- 适用范围：
  - `packages/leafergraph/src/interaction/context_menu.ts`
  - `packages/editor/src/menu/context_menu_bindings.ts`
  - `packages/editor/src/menu/context_menu_resolver.ts`
  - `packages/editor/src/ui/viewport/View.tsx`

## 1. 当前定位

当前右键菜单已经不是“某个旧 GraphViewport 文件里的临时逻辑”，而是分成了两层：

1. 主包基础设施层
   - `packages/leafergraph/src/interaction/context_menu.ts`
2. editor 菜单语义层
   - `packages/editor/src/menu/context_menu_bindings.ts`
   - `packages/editor/src/menu/context_menu_resolver.ts`

这意味着当前菜单方案的核心结论是：

- 主包负责 Leafer 菜单事件、DOM 菜单挂载和通用菜单管理器
- editor 负责节点/连线/画布三类菜单语义，以及命令总线接线

## 2. 为什么当前这样分层

这样分层可以同时满足两件事：

1. 主包不需要知道 editor 的具体命令或产品语义
2. editor 不需要重写一套 Leafer 菜单基础设施

如果重新把两层混在一起，最容易出现的问题是：

- 主包被 editor 页面细节污染
- editor 再次直接依赖底层 DOM 菜单细节

## 3. 当前文件分工

### 3.1 主包基础设施

`packages/leafergraph/src/interaction/context_menu.ts` 当前负责：

- 归一化 Leafer `pointer.menu`
- 提供菜单绑定类型
- 提供菜单上下文类型
- 管理 DOM 菜单节点与基础样式
- 暴露 `createLeaferGraphContextMenu(...)`

当前不再应该把旧的单文件 `context_menu.ts` 入口写成现行入口。

### 3.2 editor 菜单绑定

`packages/editor/src/menu/context_menu_bindings.ts` 当前负责：

- 为节点和连线生成绑定 key
- 绑定节点与连线菜单元数据
- 帮 viewport 在图元创建、删除、重建后重挂菜单

### 3.3 editor 菜单解析

`packages/editor/src/menu/context_menu_resolver.ts` 当前负责：

- 节点菜单项
- 连线菜单项
- 画布菜单项
- 节点创建分组菜单
- 菜单打开前的状态同步

它已经开始优先消费：

- `EditorCommandBus`
- `commandBus.resolveCommandState(...)`

而不是在每个菜单项里各写一套禁用判断。

### 3.4 viewport 接线

`packages/editor/src/ui/viewport/View.tsx` 当前负责：

- 创建菜单管理器
- 注册节点/连线/画布绑定
- 在 authority/session/runtime/selection 变化后同步菜单挂载

当前真实 viewport 入口是 `packages/editor/src/ui/viewport/View.tsx`，不是旧的 app 层 GraphViewport 入口。

## 4. 为什么当前仍然使用 `pointer.menu`

当前菜单基础设施仍然建立在 Leafer 的 `pointer.menu` 上，原因是：

- 它和 Leafer 命中系统一致
- 能正确感知画布、节点和连线挂载目标
- 能保留原生 `contextmenu` 的坐标与默认行为控制能力

当前文档不再把这件事描述为“历史兼容”，而是明确它就是现行基础设施选择。

## 5. 当前 API 形态

当前与菜单直接相关的公共 API 主要包括：

- `LEAFER_GRAPH_POINTER_MENU_EVENT`
- `LeaferGraphContextMenuManager`
- `createLeaferGraphContextMenu(...)`
- `LeaferGraphContextMenuBinding`
- `LeaferGraphContextMenuContext`
- `LeaferGraphContextMenuResolver`

editor 则在此基础上补自己的 resolver 和 bindings。

## 6. 当前菜单与命令总线的关系

当前菜单已经不再是“点击后直接改图”的孤立系统，而是统一走命令总线语义：

- 节点菜单消费节点命令与选区命令
- 连线菜单消费 link 命令
- 画布菜单消费 create-node、paste、fit-view 等画布命令

这让菜单、快捷键、toolbar 和后续 authority 确认更容易共享同一套操作状态。

## 7. 当前几何与坐标边界

菜单基础设施当前负责的坐标职责是：

- 接收 Leafer `pointer.menu`
- 从事件里提取页面坐标
- 把页面坐标用于 DOM 菜单定位

菜单基础设施不负责：

- 自行计算节点世界坐标布局
- 决定菜单项是否可用
- 解释具体业务动作

这些职责仍分别留在 viewport、selection、command bus 和 editor 壳层。

## 8. 当前限制

当前菜单系统已经够用，但仍有明确边界：

- 它是 editor 当前菜单实现的正式基础设施，不等于完整产品级菜单框架
- 主包只提供基础能力，不负责 editor 的所有文案和命令状态
- 菜单的复杂业务判断仍应优先沉淀到命令层，而不是回写到主包

## 9. 当前推荐做法

- 主包继续只维护菜单基础设施
- editor 继续在 `src/menu/*` 维护菜单语义
- 新增菜单动作时优先先补命令，再补菜单项
- 菜单禁用态、描述、危险标记优先从命令总线读取

## 10. 当前结论

当前右键菜单方案已经从“旧页面里的临时解析器”演进为：

- 主包：菜单基础设施
- editor：菜单绑定与菜单语义
- 命令总线：状态与执行中心

后续继续扩展右键菜单时，默认应沿着这三层分工推进，而不是重新把所有逻辑塞回 viewport 文件。
