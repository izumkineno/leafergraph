## 1. 定位

当前右键菜单能力重新归位到 `leafergraph` 主包，定位为一层可复用的交互基础设施，而不是 editor 私有模块。

职责边界如下：

1. `leafergraph` 主包负责：
   - 监听 `pointer.menu`
   - 归一化坐标与事件上下文
   - 管理画布级、节点级挂载
   - 渲染、定位、关闭 DOM 菜单
2. `editor` 负责：
   - 根据上下文决定显示哪些菜单项
   - 将菜单项动作接到真正的图命令
   - 在视口组件中把菜单管理器与图实例接起来

一句话总结：

- **主包负责菜单基础设施，editor 负责菜单语义与命令接线。**

---

## 2. 为什么放回主包

这次放回主包，核心原因是当前 `context_menu.ts` 的代码内容本身更接近宿主级能力，而不是 editor 视图私有逻辑。

它具备下面这些特征：

1. 不依赖 `Preact`
2. 不依赖 demo 节点数据
3. 不依赖 editor 面板状态
4. 只依赖 Leafer 的事件系统和通用 DOM 浮层能力

因此它更适合由主包统一提供，供 editor 或外部宿主复用。

## 3. 当前文件分工

### 3.1 `packages/leafergraph/src/context_menu.ts`

这个文件负责：

1. 监听 Leafer `PointerEvent.MENU`
2. 阻止浏览器默认右键菜单
3. 创建 `LeaferGraphContextMenuContext`
4. 维护画布与节点级挂载记录
5. 渲染 DOM 菜单
6. 处理点击外部关闭、`Escape` 关闭、窗口变化关闭

### 3.2 `packages/leafergraph/src/index.ts`

这个文件负责对外导出：

1. `LEAFER_GRAPH_POINTER_MENU_EVENT`
2. `LeaferGraphContextMenuManager`
3. `createLeaferGraphContextMenu`
4. 菜单相关的上下文、菜单项、绑定与事件类型

### 3.3 `packages/editor/src/GraphViewport.tsx`

这个文件负责：

1. 创建 `LeaferGraph`
2. 调用 `createLeaferGraphContextMenu`
3. 将画布菜单挂到 `graph.app`
4. 在 `graph.ready` 后把节点菜单挂到 `graph.getNodeView(node.id)`
5. 销毁时同步清理 graph 与 menu

## 4. 为什么仍然使用 `pointer.menu`

根据 Leafer 文档：

1. `PointerEvent.MENU`
2. 事件名是 `pointer.menu`
3. 语义等价于 HTML `contextmenu`
4. 在按下阶段触发

使用它的好处是：

1. 可以直接获取 Leafer 世界坐标 `x / y`
2. 可以继续读取 `getPagePoint()`、`getBoxPoint()`、`getLocalPoint()`
3. 可以保留 Leafer 命中的 `target`
4. 更容易区分画布菜单、节点菜单、端口菜单与局部 Widget 菜单

因此，主包菜单能力仍然建立在 Leafer 事件体系之上，而不是直接依赖原生 DOM `contextmenu`。

## 5. 当前 API 设计

主包当前对外核心类型如下：

1. `LeaferGraphContextMenuBindingTarget`
2. `LeaferGraphContextMenuBinding`
3. `LeaferGraphPointerMenuEvent`
4. `LeaferGraphContextMenuContext`
5. `LeaferGraphContextMenuActionItem`
6. `LeaferGraphContextMenuSeparatorItem`
7. `LeaferGraphContextMenuItem`
8. `LeaferGraphContextMenuResolver`
9. `LeaferGraphContextMenuOptions`

设计原则是：

1. 主包只暴露菜单真正需要的最小类型
2. 不把 Leafer 的完整内部事件结构直接泄漏到公共 API
3. 菜单层只做“事件与坐标归一化”，不做具体业务命令

---

## 6. 坐标设计

当前菜单上下文中同时提供：

1. `worldPoint`
2. `pagePoint`
3. `clientPoint`
4. `containerPoint`
5. `boxPoint`
6. `localPoint`

它们分别用于不同场景：

1. `worldPoint`
   - 适合图内部命令，例如在某个坐标创建节点
2. `clientPoint`
   - 适合 DOM `position: fixed` 菜单定位
3. `containerPoint`
   - 适合 editor 判断相对视口容器的位置
4. `boxPoint / localPoint`
   - 适合未来端口菜单和 Widget 局部菜单

---

## 7. 生命周期设计

`LeaferGraphContextMenuManager` 当前生命周期如下：

1. `bind()`
   - 注入样式
   - 创建菜单根节点
   - 挂载画布菜单
   - 安装关闭相关事件
2. `bindCanvas()`
   - 绑定画布级右键菜单
3. `bindNode()`
   - 绑定节点级右键菜单
4. `show(context, items)`
   - 规整菜单项、渲染 DOM、定位并打开菜单
5. `hide()`
   - 关闭菜单但保留根节点
6. `destroy()`
   - 解绑事件并移除菜单 DOM

为了避免同一轮右键事件同时命中画布和节点导致重复弹出，管理器内部做了事件去重。

## 8. 当前 editor 接入方式

`GraphViewport.tsx` 当前接入方式如下：

```ts
const graph = createLeaferGraph(host, { nodes });

const menu = createLeaferGraphContextMenu({
  app: graph.app,
  container: graph.container,
  resolveItems: resolveDemoMenuItems
});

graph.ready.then(() => {
  for (const node of nodes) {
    const nodeView = graph.getNodeView(node.id);
    if (!nodeView) continue;

    menu.bindNode(`node:${node.id}`, nodeView, {
      nodeId: node.id,
      nodeTitle: node.title,
      nodeType: node.type
    });
  }
});
```

这里的边界很清楚：

1. 主包提供通用菜单管理器
2. editor 负责提供菜单项解析器
3. editor 通过 `graph.getNodeView()` 把节点菜单挂到具体视图

## 9. 当前限制

现在已经支持：

1. 画布菜单
2. 节点菜单
3. DOM 菜单定位
4. 菜单关闭生命周期
5. 多挂载点事件去重

暂时还没有支持：

1. 二级子菜单
2. 键盘上下导航
3. 菜单图标系统
4. 正式命令注册中心
5. 端口级与 Widget 级局部菜单
