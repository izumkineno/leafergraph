# 架构演进与提案总览

## 文档信息

- 当前状态：前瞻性提案集中页
- 最近校对：2026-04-21
- 适用范围：`packages/authoring`、`templates/`、`example/` 及未来新增的演进提案
- 与当前事实型文档的关系：
  - 当前已落地能力，以各包 README 和事实型专题为准
  - 这里专门收口“仍在演进中、尚不应当写成当前事实”的内容

## 1. 为什么单独集中

`leafergraph` 现在已经把 README、主包深层文档和 `docs/` 里的事实型专题逐步收口成“现状优先”。

为了避免把以下两类内容继续混在一起：

- 当前源码已经成立的事实
- 仍在讨论、允许调整的演进方案

从这轮开始，所有明确属于“方案草案 / 待实现 / 演进方向”的内容统一收口到这份文档。

这样做的目的有三个：

1. 避免包 README 被未来规划污染。
2. 避免维护者把草案误读成当前实现。
3. 给未来新的 proposal 留一个固定落点，而不是继续分散到多个零散文件。

## 2. 当前仍有效的提案主题

### 2.1 作者层持续演进

当前 `@leafergraph/authoring` 已经落地为独立作者层 SDK，但这条线仍有持续演进空间。  
这部分提案现在主要用于约束后续演化方向，而不是重新定义一个还不存在的包。

当前已经稳定的结论：

- 作者层必须保持独立 workspace 包。
- 它只输出正式核心产物：
  - `NodeDefinition`
  - `NodeModule`
  - `LeaferGraphWidgetEntry`
  - `LeaferGraphNodePlugin`
- editor、模板工程、browser bundle 宿主只能适配 `authoring`，不能反过来把历史 loader / bridge / authority 负担压回包内。

当前仍然保留为演进约束的方向：

- 节点作者层与 Widget 作者层继续保持对称体验。
- 作者层尽量覆盖当前 `NodeLifecycle` 与 Widget lifecycle 的稳定子集。
- 作者层继续优先服务模板和外部生态，而不是仓库内部专用壳层。
- 需要新增更高阶作者体验时，优先扩展作者层 API，而不是回流到 `leafergraph` 主包。

### 2.2 模板与宿主接线的长期边界

模板、bundle loader 和外部宿主的长期原则保持不变：

- 模板工程负责输出对外消费友好的产物。
- 宿主负责 bundle 装配、页面壳层、菜单和外围协议接线。
- `authoring` 不承担 bundle catalog、authority session、OpenRPC、editor 面板等壳层逻辑。

当后续需要扩模板时，优先沿这条链路思考：

1. 先确认这是不是作者层能力。
2. 如果不是，再确认它属于模板本身还是外部宿主。
3. 不要为了省事把模板约定重新塞回主包或作者层。

### 2.3 已批准的 package split

当前已经批准、并进入执行阶段的 proposal，是把正式包拆到两条稳定目录：

- `packages/core/*`
- `packages/extensions/*`

这轮拆分的目标不是重新发明 API，而是把“真源核心”和“宿主扩展”物理落盘到更清晰的目录结构里，同时保留 `leafergraph` 作为兼容主包。

#### 当前包到目标包的映射

| 当前包 | 目标包 |
| --- | --- |
| `@leafergraph/node` | `@leafergraph/core/node` |
| `@leafergraph/execution` | `@leafergraph/core/execution` |
| `@leafergraph/contracts` | `@leafergraph/core/contracts` |
| `@leafergraph/config` | `@leafergraph/core/config` |
| `@leafergraph/theme` | `@leafergraph/core/theme` |
| `@leafergraph/widget-runtime` | `@leafergraph/core/widget-runtime` |
| `@leafergraph/basic-kit` | `@leafergraph/core/basic-kit` |
| `@leafergraph/context-menu` | `@leafergraph/extensions/context-menu` |
| `@leafergraph/context-menu-builtins` | `@leafergraph/extensions/context-menu-builtins` |
| `@leafergraph/shortcuts` | `@leafergraph/extensions/shortcuts` |
| `@leafergraph/undo-redo` | `@leafergraph/extensions/undo-redo` |
| `@leafergraph/authoring` | `@leafergraph/extensions/authoring` |
| `leafergraph` | `leafergraph`（兼容主包，继续保留） |

#### 目标

- 让 core foundation / core runtime / extensions 的目录边界和文档边界保持一致。
- 让 `README`、`example/`、`templates/` 在讲未来结构时可以直接引用稳定 target package。
- 让 `leafergraph` 的兼容策略只在主包内收口，不把历史聚合重新扩散回 core 包。

#### 非目标

- 不在这轮 proposal 里扩大主包职责，让 `leafergraph` 重新变回全量 re-export 聚合。
- 不把作者层重新塞回主包或模板壳层。
- 不把“拆分后的目标结构”误写成“当前已经落地事实”。

#### 文档与样例的固定边界

这轮 split 对文档和样例的要求固定为：

- `README.md`
  - 讲清楚当前真实包边界
  - 同时给出“当前包 → 目标包”的映射
- `docs/`
  - 当前事实仍以源码为准
  - 演进中的 split 只在 proposal 文档里收口
- `example/`
  - 重点证明主包兼容层怎样装配 core + extensions
- `templates/`
  - 重点证明模板仍围绕 `authoring` + core 真源组织，不依赖历史壳层

## 3. 作者层提案的当前建议写法

如果后续继续在这份总览里追加作者层相关 proposal，建议固定回答下面这些问题：

- 这个 proposal 服务谁：节点作者、Widget 作者、模板维护者，还是宿主集成者？
- 它会新增什么正式产物：类型、helper、基类、bundle 约定，还是只是维护约束？
- 它和 `@leafergraph/node`、`@leafergraph/contracts`、`leafergraph` 的边界是什么？
- 它是否会让作者层反向依赖 editor 或宿主壳层？

只要其中任意一条回答不清楚，就不应该把它写成正式 README 内容。

## 4. 后续提案进入这里的规则

后续新增 proposal 时，统一按这套结构追加：

1. 提案标题
2. 当前状态
3. 目标与非目标
4. 包边界 / 宿主边界
5. 与当前已落地能力的关系
6. 暂不落地的原因

同时遵守两个固定约束：

- 不在这里维护“现状说明”，现状回到 README 或事实型专题。
- 不在这里写碎片 TODO；这里只收口足够完整、能独立阅读的提案。

## 5. 当前结论

对今天的维护者来说，这份文档最重要的作用不是告诉你“现在系统怎么工作”，而是提醒你：

- README 和事实型专题讲当前真相。
- 这份总览讲未来演进约束。
- 当二者冲突时，当前源码和 README 优先。
