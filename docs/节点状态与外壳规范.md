- 适用范围：`packages/node`、`packages/authoring`、`packages/execution`、`packages/leafergraph`

## 1. 状态分类

Node Shell 只表达两类稳定状态：

1. 执行态
   - `idle`
   - `running`
   - `success`
   - `error`
2. 外壳态
   - `selected`
   - `collapsed`
   - `missing`
   - `long-task active`

下面这些不是 shell 常驻状态，只是刷新原因：

- `input-values`
- `widget-action`
- `widget-value`
- `connections`
- `updated`

## 2. 视觉职责

### 2.1 Signal Light

左上角只保留一组旧信号灯，不再渲染新的 `signal cluster`、badge 或 activity dot。

信号灯职责固定为：

- 只表示执行态
  - `idle`：默认灰
  - `running`：运行色
  - `success`：成功色
  - `error`：错误色
- 不承载：
  - `selected`
  - `collapsed`
  - `missing`
  - `long-task active`

`signalButton` 继续作为折叠命中区；缺失节点不提供折叠交互。

### 2.2 Selected Ring

`selected ring` 只表示选中态，不再承担：

- hover
- resizing
- running

### 2.3 Progress Ring

`progress ring` 是最外层 rounded-rect 外环，只表示长任务进度：

- 有 `execution.state.progress` 时显示 determinate 进度
- `shell.longTask === true` 且节点处于 `running`、但没有 progress 时显示 indeterminate 运行环
- `idle / success / error` 时清空 progress ring
- 路径必须沿节点卡片外轮廓绘制，使用精确 rounded-rect 路径，不使用采样折线近似

`selected ring` 和 `progress ring` 必须可同时存在。

### 2.4 其他状态表达

以下状态不再通过第二套顶部灯位表达：

- `collapsed`
  - 继续由折叠后的节点结构和现有交互表达
- `missing`
  - 继续由 missing palette、标题和内容文案表达
- `long-task active`
  - 由 `running` 信号灯颜色 + `progress ring` 共同表达

## 3. 作者侧声明

节点作者可以在节点定义里声明：

```ts
shell: {
  longTask: true
}
```

可用入口：

- `NodeDefinition.shell.longTask`
- `DevNodeMeta.shell.longTask`

语义固定为：

- 这是一个可能长时间处于 `running` 的节点
- 仅用于“没有精确 progress 时是否显示 indeterminate 外环”
- 不是业务状态文案入口

## 4. Progress 投影

正式进度来源固定为执行反馈：

```ts
state: {
  status: "running",
  progress: 0.42
}
```

规则固定为：

- `progress` 范围是 `0..1`
- 只有 `status === "running"` 时有效
- 进入 `success / error / idle` 后应清空 `progress`
- v1 不通过 `properties`、`data` 或 widget 值隐式读取进度
- 对单飞长任务节点，新的忙碌输入不能重置当前 active task 的 `progress`
- 忙碌输入如果被节点丢弃，shell 仍应保持当前 `running/progress`，直到 active task 真正结束

## 5. 长任务与排队规范

### 5.1 Delay 类长任务节点

`events/delay` 是当前规范里的参考实现，行为固定为：

- 单飞执行
  - 同一时刻只允许一个 active ticket
- 忙碌时默认丢弃新输入
  - 不追加内部 pending 队列
  - 不重注册 timer
  - 不重置当前 progress ring
- progress ring 只跟随当前 active ticket
  - 输入频率、外部 backlog、overflow 都不能重置它

推荐状态文案包括：

- `READY`
- `RUNNING`
- `FIRED`
- `DROPPED`

### 5.2 Queue 节点

需要排队时，应显式使用 `events/queue`，而不是把排队逻辑塞进长任务节点内部。

`events/queue` 的规范行为：

- 输入
  - `push`
  - `release`
  - `capacity?`
- 输出
  - `next`
  - `overflow`
- 运行模型
  - `push` 到来时，如果当前未等待 release 且队列为空，立即放行一项并进入 `awaitingRelease=true`
  - 其余输入进入 FIFO 队列
  - 队列满时丢弃新输入，并从 `overflow` 输出原始 payload
  - `release` 到来时，放行下一项；如果队列为空则退出等待态

`events/queue` 不使用 long-task progress ring，它的这些信息应通过 widget / 状态文案表达：

- `queued`
- `capacity`
- `dropped`
- `awaiting release`

## 6. Shell 与 Widget 的边界

适合放在 shell 的内容：

- 执行态
- 选中态
- 折叠态
- 缺失态
- 长任务是否仍在运行
- 粗粒度进度

适合放在 widget / 内容区的内容：

- 业务阶段名
- 多段流程细节
- 自定义百分比标签
- backlog / queued / overflow / dropped 计数
- 波形、频谱、日志、性能读数
- 任何需要业务上下文才能理解的状态

原则是：

- shell 负责“全图里一眼可扫到的稳定信号”
- widget 负责“节点自身的业务细节”

## 7. Reduced Motion

当启用 `prefers-reduced-motion: reduce` 且宿主选择遵循时：

- 不播放 indeterminate progress ring 的 sweep 动效
- 改为静态 partial ring
- 仍保留运行中的可见标记，不把状态直接隐藏

## 8. 节点作者建议

推荐顺序：

1. 先判断节点是否真的会长期停留在 `running`
2. 只有需要时再声明 `shell.longTask`
3. 如果外部 runtime 能提供真实进度，再投影 `execution.state.progress`
4. 如果节点需要排队，优先使用显式 `Queue` 或等价握手节点，不要让长任务节点既执行又隐式积压
5. 业务文字和细节继续放在 widget，不要额外设计第二套顶部状态灯
