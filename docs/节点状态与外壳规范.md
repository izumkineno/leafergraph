## 1. 状态分类

Node Shell 只表达两类稳定状态：

1. 执行态
   - `idle`
   - `running`
   - `success`
   - `error`
2. 外壳态
   - `selected`
   - `collapsed`
   - `missing`
   - `long-task active`

下面这些不是 shell 常驻状态，只是刷新原因：

- `input-values`
- `widget-action`
- `widget-value`
- `connections`
- `updated`

## 2. 视觉职责

### 2.1 Signal Cluster

左上角 `signal cluster` 负责汇总稳定状态：

- 核心灯只表示执行态
  - `idle`：默认灰
  - `running`：运行色
  - `success`：成功色
  - `error`：错误色
- badge 表示壳体补充状态
  - `C`：当前节点已折叠
  - `!`：当前节点类型缺失
- activity dot 表示长任务仍在运行

`signalButton` 继续是折叠命中区；缺失节点不提供折叠交互。

### 2.2 Selected Ring

`selected ring` 只表示选中态，不再承担：

- hover
- resizing
- running

### 2.3 Progress Ring

`progress ring` 是最外层 rounded-rect 外环，只表示长任务进度：

- 有 `execution.state.progress` 时显示 determinate 进度
- `shell.longTask === true` 且节点处于 `running`、但没有 progress 时显示 indeterminate 运行环
- `idle / success / error` 时清空 progress ring

`selected ring` 和 `progress ring` 必须可同时存在。

## 3. 作者侧声明

节点作者可以在节点定义里声明：

```ts
shell: {
  longTask: true
}
```

可用入口：

- `NodeDefinition.shell.longTask`
- `DevNodeMeta.shell.longTask`

语义固定为：

- 这是一个可能长时间处于 `running` 的节点
- 仅用于“没有精确 progress 时是否显示 indeterminate 外环”
- 不是业务状态文案入口

## 4. Progress 投影

正式进度来源固定为执行反馈：

```ts
state: {
  status: "running",
  progress: 0.42
}
```

规则固定为：

- `progress` 范围是 `0..1`
- 只有 `status === "running"` 时有效
- 进入 `success / error / idle` 后应清空 `progress`
- v1 不通过 `properties`、`data` 或 widget 值隐式读取进度

## 5. Shell 与 Widget 的边界

适合放在 shell 的内容：

- 执行态
- 选中态
- 折叠态
- 缺失态
- 长任务是否仍在运行
- 粗粒度进度

适合放在 widget / 内容区的内容：

- 业务阶段名
- 多段流程细节
- 自定义百分比标签
- 波形、频谱、日志、性能读数
- 任何需要业务上下文才能理解的状态

原则是：

- shell 负责“全图里一眼可扫到的稳定信号”
- widget 负责“节点自身的业务细节”

## 6. Reduced Motion

当启用 `prefers-reduced-motion: reduce` 且宿主选择遵循时：

- 不播放 indeterminate progress ring 的 sweep 动效
- 改为静态 partial ring
- 仍保留运行中的可见标记，不把状态直接隐藏

## 7. 节点作者建议

推荐顺序：

1. 先判断节点是否真的会长期停留在 `running`
2. 只有需要时再声明 `shell.longTask`
3. 如果外部 runtime 能提供真实进度，再投影 `execution.state.progress`
4. 业务文字和细节继续放在 widget，不要塞进 shell badge
