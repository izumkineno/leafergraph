# 外部节点包接入方案

## 文档信息

- 当前状态：现状优先，保留接入建议
- 最近校对：2026-03-23
- 适用范围：`packages/node`、`packages/leafergraph`、外部 bundle 宿主、模板工程

## 1. 结论先行

当前外部节点接入应明确分成两条路径：

1. 正式包接入
   - 基于 `NodeModule`
   - 通过 `installNodeModule(...)`、主包插件或主包注册入口进入系统
2. bundle 宿主装载接入
   - 基于 `demo / node / widget` bundle
   - 通过宿主 loader 和 `LeaferGraphEditorBundleBridge.registerBundle(...)` 进入系统

当前推荐优先级是：

- 长期能力、对外复用、稳定生态：优先正式包接入
- 本地联调、快速验证、演示交付：再补 browser bundle 接入

## 2. 当前问题定义

外部节点接入当前要同时解决三件事：

1. 节点定义怎样进入统一注册表。
2. widget renderer 怎样进入统一 widget registry。
3. 外部宿主在不源码直连外部工程的情况下，怎样装入本地或远端 bundle。

如果不区分这三件事，最容易出现的问题就是：

- 把 bundle bridge 误当成主包长期 API
- 把宿主本地联调协议写成正式分发协议
- 把节点模块、widget 注册和蓝图文档混成一个“万能插件”

## 3. 当前正式接入模型

### 3.1 `NodeModule` 是当前正式节点扩展入口

当前推荐的模型中心仍然是：

- `NodeDefinition`
- `NodeModule`
- `NodeRegistry`
- `installNodeModule(...)`

这条路径的特点是：

- 语义清晰
- 不依赖浏览器脚本桥
- 可以稳定服务主包和后续外部应用接入

### 3.2 Widget 走统一注册入口

当前 widget 的正式宿主入口已经收口为：

- `registerWidget(entry)`

因此当前外部插件应默认理解为：

- 节点定义进入节点注册表
- widget 定义进入 widget registry
- 两者在宿主层汇合，而不是互相替代

### 3.3 browser bundle 只是额外接入形式

当前 browser bundle 的职责是：

- 帮外部作者在浏览器内快速导入 node / widget / demo
- 帮 authority 推送前端 bundle
- 帮模板工程提供本地联调与演示产物

它不是用来替代 `NodeModule` 或正式包分发的唯一方案。

## 4. 当前推荐的接入边界

### 4.1 模型层和宿主装载层边界

必须持续区分：

| 层 | 当前正式入口 |
| :--- | :--- |
| 模型层 | `NodeDefinition`、`NodeModule`、`installNodeModule(...)` |
| widget 宿主层 | `registerWidget(entry)` |
| bundle 宿主层 | `EditorBundleManifest`、`LeaferGraphEditorBundleBridge.registerBundle(...)` |

### 4.2 authority 与插件接入边界

当前 authority 可以同步：

- 正式 `GraphDocument`
- `GraphDocumentDiff`
- `runtimeFeedback`
- `frontendBundles.sync`

但这不改变插件接入的长期边界：

- authority 同步的是会话和远端来源
- 插件接入定义的是节点与组件怎样进入系统

## 5. 当前推荐目录与产物模型

### 5.1 正式包推荐结构

当前正式包侧建议维持类似结构：

```text
your-node-package/
  src/
    index.ts
    module.ts
    nodes/
    widgets/
```

其中：

- `module.ts` 负责导出 `NodeModule`
- `nodes/` 放节点定义
- `widgets/` 放 widget entry 或配套逻辑
- `index.ts` 作为包公共入口

### 5.2 browser bundle 产物推荐结构

若需要 browser bundle 导入，则再额外补 browser 产物，例如：

```text
dist/browser/
  node.iife.js
  widget.iife.js
  demo.iife.js
  demo-alt.iife.js
```

当前这条链推荐直接使用：

- `templates/misc/authoring-browser-plugin-template`

## 6. 当前模板与构建建议

### 6.1 当前模板定位

当前模板工程不是“唯一正式分发方式”，而是：

- 外部作者样例
- browser bundle 样例
- ESM + IIFE 双产物样例

### 6.2 构建策略建议

当前推荐保持：

- 正式 ESM 包构建
- browser IIFE bundle 构建

这样可以同时服务：

- 主包或其它工程 `import`
- 支持这些 bundle 的宿主或 authority 前端 bundle 同步

### 6.3 命名建议

当前文档不再推荐使用 `@demo/*` 作为默认示例包名，因为这会继续强化“demo 就是正式生态”的错觉。

当前更推荐使用中性命名，例如：

- `@example/math-nodes`
- `@sample/image-tools`
- `@your-org/vision-nodes`

## 7. 当前 bundle 语义

### 7.1 `node` bundle

负责：

- 节点插件安装
- 可选 `quickCreateNodeType`

### 7.2 `widget` bundle

负责：

- widget renderer
- 组件配套逻辑

### 7.3 `demo` bundle

负责：

- 一份正式 `GraphDocument`

当前最佳实践仍然是把三者分开，而不是做一个“大而全”单 bundle。

## 8. 当前最重要的实现约束

### 8.1 主包持有唯一注册中心

当前必须坚持：

- 节点注册表由宿主持有
- widget registry 由宿主持有
- 外部包只提供定义和安装逻辑

不要再回到“外部包自己持有一份 registry 再想办法和宿主同步”的方向。

### 8.2 bundle bridge 不进入主包长期 API

当前必须坚持：

- `LeaferGraphEditorBundleBridge.registerBundle(...)` 只属于宿主装载层
- 主包长期公共 API 不依赖它

### 8.3 继续让节点快照与图操作保持正式化

当前外部节点若要接 authority 链，应默认满足：

- 节点快照可序列化
- 节点更新可表达成 `GraphOperation`
- 运行态变化可映射到 `RuntimeFeedbackEvent`

## 9. 当前接入顺序建议

### 9.1 面向正式生态

1. 先写节点定义和模块
2. 再写 widget entry
3. 再在宿主侧安装模块与 widget
4. 如需 authority 联调，再校准序列化、操作和反馈语义

### 9.2 面向 bundle 联调

1. 基于模板产出 `widget.iife.js`
2. 再产出 `node.iife.js`
3. 最后产出 `demo.iife.js`
4. 在支持这些 bundle 的宿主或 authority `frontendBundles.sync` 中验证装载链

## 10. 当前不推荐的做法

- 不要把 browser bundle 当成唯一正式发布格式
- 不要把 `demo` 槽位命名扩散成主包长期模型命名
- 不要让外部包自己持有 registry
- 不要把某个外部应用专属协议直接写进核心节点模型
- 不要继续在文档里把旧 demo registry 描述为现行主线

## 11. 当前结论

当前外部节点接入的最合理表达是：

- 正式模型与宿主接入，走 `NodeModule + registerWidget(entry)`
- 宿主本地联调与远端前端扩展，走 bundle loader
- authority 会消费这些结果，但不会替代这两条接入边界

只要持续守住这三条边界：

- 模型层
- 宿主层
- 宿主装载层

后续无论是模板工程、authority 同步还是外部应用接入，都不会把架构重新拉回旧 demo 时代。
