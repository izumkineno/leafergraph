# 开发者友好节点与 Widget 作者层独立包方案

## 文档信息

- 当前状态：方案草案，待实现
- 最近校对：2026-03-25
- 适用范围：未来独立作者层包、`packages/node`、`packages/leafergraph`、`templates/`、内部示例
- 直接前提：
  1. 尽量覆盖现有 `NodeLifecycle`
  2. 这个草案本身与 editor 无关
  3. 它是一个独立包，面向节点 / widget 开发者与内部示例
  4. 若 editor、模板工程或其他消费方要使用它，只能由外部去适配这个包，不能反过来把历史 loader / bundle 负担压回包内

## 1. 定位

这份方案的目标不是修改 core，也不是新增 editor 专属协议，而是补一个独立作者层包。

这个包的定位是：

- 面向节点开发者
- 面向 widget 开发者
- 面向仓库内的内部示例
- 用更友好的类式 authoring 体验，屏蔽一部分底层接线细节
- 最终仍然输出当前核心库已经接受的正式注册对象

一句话说，这是一层独立的 authoring SDK，不是新的底层节点系统，也不是 editor 子系统。

## 2. 它和外部消费方的关系

这份方案必须明确和 editor、模板工程、bundle wrapper 解耦。

### 2.1 包本身不依赖 editor

这个包本身不应：

- 依赖 `packages/editor`
- 依赖 loader runtime
- 依赖 authority session
- 依赖 editor 的 bundle bridge
- 假设一定运行在 editor 页面里

它应该只依赖：

- `@leafergraph/node`
- `leafergraph`
- `leafer-ui`

### 2.2 外部消费方只能适配它

这个包产出的结果应该能被几类宿主消费：

1. 直接使用 `leafergraph` 的宿主
   - 例如 `graph.use(plugin)`
2. 仓库内内部示例
   - 例如模板工程、联调样例、演示包
3. 其他外部消费层
   - 例如模板工程、自定义 wrapper、未来其他宿主

因此这里必须写死一个方向：

- 它独立存在
- 它的公共 API 只围绕正式核心产物组织
- editor、模板工程或其他消费方若要使用它，必须在包外自行适配

这意味着：

- authoring 包不能因为“未来可能进入 editor”而带上 loader manifest、bundle bridge、`node-json`、`demo / node / widget` 槽位命名、`quickCreateNodeType` 或 script/IIFE 包装语义
- 也不能把“兼容现有 editor 装载链”写成这个包的设计目标
- 正确方向只能是“editor 兼容这个包”，不能反过来要求这个包兼容 editor 历史负担

## 3. 当前源码约束

这套方案必须对齐当前已经存在的正式接口，而不是绕开它们另起一套 runtime。

### 3.1 节点生命周期必须尽量覆盖现有 `NodeLifecycle`

当前 `@leafergraph/node` 已有正式生命周期集合：

- `onCreate`
- `onConfigure`
- `onSerialize`
- `onExecute`
- `onPropertyChanged`
- `onInputAdded`
- `onOutputAdded`
- `onConnectionsChange`
- `onAction`
- `onTrigger`

当前没有正式 `destroy` / `dispose` 生命周期。

因此作者层包应尽量映射上述生命周期，但不能凭空承诺一个 core 并不存在的 destroy 回调。

但这里必须继续区分两件事：

- `NodeLifecycle` 已经声明了哪些钩子
- 当前宿主实际上稳定调度了哪些钩子

作者层 v1 建议按下面这张矩阵设计：

| 生命周期                | core 已声明 | 当前宿主已稳定调度 | 作者层 v1 建议                    |
| :---------------------- | :---------- | :----------------- | :-------------------------------- |
| `onCreate`            | 是          | 是                 | 正式支持                          |
| `onConfigure`         | 是          | 是                 | 正式支持                          |
| `onSerialize`         | 是          | 是                 | 正式支持                          |
| `onExecute`           | 是          | 是                 | 正式支持                          |
| `onInputAdded`        | 是          | 是                 | 正式支持                          |
| `onOutputAdded`       | 是          | 是                 | 正式支持                          |
| `onConnectionsChange` | 是          | 是                 | 正式支持                          |
| `onAction`            | 是          | 是                 | 正式支持                          |
| `onPropertyChanged`   | 是          | 否                 | v1 只保留映射位，不写成已落地能力 |
| `onTrigger`           | 是          | 否                 | v1 只保留映射位，不写成已落地能力 |

另外还要写清一个当前实现事实：

- `onExecute` / `onAction` / `onTrigger` 目前都应按同步回调理解
- 当前宿主不会把它们当成 Promise 生命周期去等待
- 因此作者层 v1 不应把异步返回值写成稳定合同
- 如果未来 core 明确引入异步执行语义，再单独扩 `async authoring` 能力

### 3.2 Widget 已经有正式宿主接口

当前 widget 侧正式接口已经成立：

- 数据定义：`LeaferGraphWidgetEntry`
- 渲染协议：`renderer(...)`
- 结构化生命周期：`LeaferGraphWidgetLifecycle`
- 值回写：`setValue(...)`、`commitValue(...)`
- 节点动作回抛：`emitAction(...)`

所以 widget 作者层应该建立在现有 widget lifecycle 之上，而不是自己再造一套脱离宿主的渲染协议。

### 3.3 独立包只输出正式核心注册对象

这个包最终应该输出：

- `NodeDefinition`
- `NodeModule`
- `LeaferGraphWidgetEntry`
- `LeaferGraphNodePlugin`

而不是输出 editor 专属对象。

也就是说，真正进入核心库的仍然是当前正式接口；作者层包只是帮助开发者更容易生成这些正式对象。

如果未来要补 demo 预设辅助，应另开一轮方案，单独判断它是否进入公共 API；本轮 v1 不把它并入 node / widget authoring 的首批职责。

## 4. 总体设计

推荐拆成三层：

| 层               | 主要职责                            | 对外输出                                                                                     |
| :--------------- | :---------------------------------- | :------------------------------------------------------------------------------------------- |
| 作者层           | 节点类、widget 类、类型化上下文     | `BaseNode`、`BaseWidget`                                                                 |
| adapter / SDK 层 | 生命周期映射、状态持有、plugin 组装 | `defineAuthoringNode(...)`、`defineAuthoringWidget(...)`、`createAuthoringPlugin(...)` |
| 正式接入层       | 核心库当前已支持的正式入口          | `NodeDefinition`、`NodeModule`、`LeaferGraphWidgetEntry`、`LeaferGraphNodePlugin` |

最重要的边界是：

- core 不改模型
- 外部消费层自己做适配
- 独立包只负责降低开发门槛

### 4.1 包形态与依赖策略

为了避免后续外部消费层和宿主装配各自再推一套包边界，建议在方案里先把包形态写清楚：

- 仓库内落点
  - 建议新增 `packages/authoring`
- 对外包名
  - 建议使用 `@leafergraph/authoring`
- 导出形态
  - 默认只提供 ESM
  - 不额外补 CJS 兼容出口
- 依赖策略
  - `@leafergraph/node`
  - `leafergraph`
  - `leafer-ui`
  - 这三者建议作为 `peerDependencies` 暴露给外部宿主
  - 仓库内开发与模板工程可同时在 `devDependencies` 中安装，方便类型检查和构建

这样做的目的不是增加发布复杂度，而是避免：

- 宿主和作者层各自装出两份 `leafergraph`
- 模板工程和正式包在依赖关系上分叉
- 后续为了兼容重复实例再补一层隐藏桥接

## 5. 节点作者层 API

### 5.1 `DevNodeMeta`

作者层节点类建议通过静态 `meta` 暴露节点定义元信息：

```ts
import type {
  NodePropertySpec,
  NodeSlotSpec,
  NodeWidgetSpec
} from "@leafergraph/node";

export interface DevNodeMeta {
  type: string;
  title: string;
  category?: string;
  description?: string;
  keywords?: string[];
  size?: [number, number];
  resize?: {
    enabled?: boolean;
    lockRatio?: boolean;
    minWidth?: number;
    minHeight?: number;
    maxWidth?: number;
    maxHeight?: number;
    snap?: number;
  };
  inputs?: NodeSlotSpec[];
  outputs?: NodeSlotSpec[];
  properties?: NodePropertySpec[];
  widgets?: NodeWidgetSpec[];
}
```

### 5.2 `DevNodeContext`

作者层上下文建议同时提供：

- 正式 `NodeRuntimeState`
- 类型化 `props`
- 类型化运行时 `state`
- 执行上下文
- 类型化输入输出访问
- `data` 读写
- widget 值读写

```ts
import type {
  NodeApi,
  NodeRuntimeState
} from "@leafergraph/node";
import type { LeaferGraphExecutionContext } from "leafergraph";

type NodeProps = Record<string, unknown>;
type NodeInputs = Record<string, unknown>;
type NodeOutputs = Record<string, unknown>;
type NodeState = Record<string, unknown>;

export interface DevNodeContext<
  P extends NodeProps,
  I extends NodeInputs,
  O extends NodeOutputs,
  S extends NodeState
> {
  node: NodeRuntimeState;
  api: NodeApi;
  props: P;
  data: Record<string, unknown>;
  state: S;
  execution?: LeaferGraphExecutionContext;

  getInput<K extends keyof I & string>(name: K): I[K] | undefined;
  getInputAt(slot: number): unknown;
  setOutput<K extends keyof O & string>(name: K, value: O[K]): void;
  setOutputAt(slot: number, value: unknown): void;

  getWidget(name: string): unknown;
  setWidget(name: string, value: unknown): void;

  setProp<K extends keyof P & string>(name: K, value: P[K]): void;
  getData<T = unknown>(name: string): T | undefined;
  setData(name: string, value: unknown): void;
}
```

### 5.3 `BaseNode`

作者层节点类应尽量覆盖现有 `NodeLifecycle`：

```ts
import type {
  NodeSerializeResult,
  NodeSlotSpec,
  SlotDirection
} from "@leafergraph/node";

export abstract class BaseNode<
  P extends NodeProps,
  I extends NodeInputs,
  O extends NodeOutputs,
  S extends NodeState = {}
> {
  static meta: DevNodeMeta;

  createState?(): S;

  onCreate?(ctx: DevNodeContext<P, I, O, S>): void;
  onConfigure?(
    data: NodeSerializeResult,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
  onSerialize?(
    data: NodeSerializeResult,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
  onExecute?(ctx: DevNodeContext<P, I, O, S>): void;
  onPropertyChanged?(
    name: keyof P & string,
    value: unknown,
    prevValue: unknown,
    ctx: DevNodeContext<P, I, O, S>
  ): boolean | void;
  onInputAdded?(
    input: NodeSlotSpec,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
  onOutputAdded?(
    output: NodeSlotSpec,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
  onConnectionsChange?(
    type: SlotDirection,
    slot: number,
    connected: boolean,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
  onAction?(
    action: string,
    param: unknown,
    options: Record<string, unknown> | undefined,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
  onTrigger?(
    action: string,
    param: unknown,
    options: Record<string, unknown> | undefined,
    ctx: DevNodeContext<P, I, O, S>
  ): void;
}
```

### 5.4 `DevNodeClass`

为了让 `defineAuthoringNode(...)` / `createAuthoringModule(...)` 的输入合同可实现，文档里还要把节点类构造类型补齐：

```ts
export interface DevNodeClass<
  P extends NodeProps = {},
  I extends NodeInputs = {},
  O extends NodeOutputs = {},
  S extends NodeState = {}
> {
  new (): BaseNode<P, I, O, S>;
  readonly meta: DevNodeMeta;
}
```

v1 建议同时写死这条构造合同：

- 节点作者类默认必须是零参数构造
- 不在 v1 引入依赖注入式 constructor 参数
- 宿主能力统一通过 `DevNodeContext` 提供，而不是通过 `new BaseNode(...)` 传入

### 5.5 节点侧合同

节点作者层需要把这几条合同写死：

- `props`
  - 对应 `node.properties`
  - 代表正式配置
- `data`
  - 对应 `node.data`
  - 代表可持久化扩展数据
- `state`
  - 只属于 SDK 运行时
  - 不参与序列化
  - 节点重建、整图替换或运行时重装后应重新创建
- `execution`
  - 只在 `onExecute(...)` 这类执行阶段保证可用
  - 对应当前主包已经存在的正式 `LeaferGraphExecutionContext`
  - 至少应保留 `source`、`runId`、`entryNodeId`、`stepIndex`、`payload`
- `getInput(name)` / `setOutput(name, value)`
  - 建立在唯一 slot 名之上
  - SDK 应明确禁止同一节点定义里出现重名输入或重名输出
- `setWidget(name, value)`
  - 只负责当前运行时 widget 值
  - 不自动定义宿主如何持久化它
- `onPropertyChanged`
  - 当前 core 只有声明，宿主尚未形成统一调度链
  - v1 可保留映射接口，但文档必须明确它不是当前稳定能力
- `onTrigger`
  - 当前与 `onPropertyChanged` 一样，不能在 v1 文档里承诺为“已稳定可用”

### 5.6 节点实例与 state 持有边界

作者层如果要提供类式 `BaseNode`，就必须把“实例和 state 到底谁持有”写清楚。

v1 建议明确为：

- `DevNodeClass` 默认按“每个节点运行时实例一份 `new NodeCtor()`”创建
- `BaseNode` 实例与 `state` 由 authoring SDK adapter 层持有
- 默认以 `node.id` 作为实例映射键
- `state` 生命周期从节点进入当前运行时开始
- `state` 生命周期在下面几种场景结束：
  - 节点被删除
  - 整图替换
  - 运行时重建
  - 节点类型切换导致实例重建

同时还要写死两条限制：

- 因为当前 core 没有正式节点 destroy 生命周期
  - v1 不向节点作者承诺 `dispose()` / `destroy()`
- 节点作者不应把必须显式释放的外部资源绑死在 `BaseNode`
  - 这类需求要么交给 widget lifecycle
  - 要么等未来 core 真正引入节点销毁钩子后再升级

## 6. Widget 作者层 API

### 6.1 `DevWidgetMeta`

```ts
import type { NodeWidgetSpec } from "@leafergraph/node";

export interface DevWidgetMeta {
  type: string;
  title?: string;
  description?: string;
  normalize?(value: unknown, spec?: NodeWidgetSpec): unknown;
  serialize?(value: unknown, spec?: NodeWidgetSpec): unknown;
}
```

### 6.2 `DevWidgetContext`

Widget 作者层上下文应把常用字段直接抬平，不要求作者每次回退到 `raw`：

```ts
import type {
  LeaferGraphWidgetBounds,
  LeaferGraphWidgetEditingContext,
  LeaferGraphWidgetRendererContext,
  LeaferGraphWidgetThemeContext
} from "leafergraph";
import type { NodeRuntimeState, NodeWidgetSpec } from "@leafergraph/node";
import type { Group } from "leafer-ui";

type WidgetState = Record<string, unknown>;

export interface DevWidgetContext<TValue> {
  raw: LeaferGraphWidgetRendererContext;
  ui: typeof import("leafer-ui");
  group: Group;
  node: NodeRuntimeState;
  widget: NodeWidgetSpec;
  widgetIndex: number;
  value: TValue;
  bounds: LeaferGraphWidgetBounds;
  theme: LeaferGraphWidgetThemeContext;
  editing: LeaferGraphWidgetEditingContext;
  setValue(nextValue: TValue): void;
  commitValue(nextValue?: TValue): void;
  requestRender(): void;
  emitAction(
    action: string,
    param?: unknown,
    options?: Record<string, unknown>
  ): boolean;
}
```

### 6.3 `BaseWidget`

```ts
export abstract class BaseWidget<
  TValue = unknown,
  TState extends WidgetState = {}
> {
  static meta: DevWidgetMeta;

  mount?(ctx: DevWidgetContext<TValue>): TState | void;
  update?(
    state: TState | void,
    ctx: DevWidgetContext<TValue>,
    nextValue: TValue
  ): void;
  destroy?(state: TState | void, ctx: DevWidgetContext<TValue>): void;
}
```

### 6.4 `DevWidgetClass`

和节点侧一样，文档里还需要把 widget 作者类的构造类型补齐：

```ts
export interface DevWidgetClass<
  TValue = unknown,
  TState extends WidgetState = {}
> {
  new (): BaseWidget<TValue, TState>;
  readonly meta: DevWidgetMeta;
}
```

v1 建议同时写死这条构造合同：

- widget 作者类默认必须是零参数构造
- 不在 v1 引入宿主注入式 constructor 参数
- 宿主能力统一通过 `DevWidgetContext` 提供

### 6.5 Widget 侧合同

Widget 作者层的核心原则是：

- 继续走当前正式 `LeaferGraphWidgetLifecycle`
- 继续尊重 `setValue(...)` / `commitValue(...)` 的语义差异
- 不把 DOM、全局菜单、编辑器单例等重新塞回 widget 作者代码
- 只负责把交互交给宿主，不负责定义宿主一定怎样持久化提交结果

同时建议补一条实例边界，避免 widget 作者误以为这里是单例：

- `DevWidgetClass` 默认按“每个挂载中的 widget render instance 一份 `new WidgetCtor()`”创建
- 该实例的生命周期与当前 widget mount / update / destroy 链一致
- 不应把跨节点、跨图的共享可变状态直接塞进 widget 类实例

## 7. 统一 SDK API

既然目标是“简化节点和 widget 开发流程”，顶层入口不应要求开发者手工拼两套低层 API。

推荐最少提供这四组 API：

```ts
export function defineAuthoringNode(
  NodeCtor: DevNodeClass
): NodeDefinition;

export function defineAuthoringWidget(
  WidgetCtor: DevWidgetClass
): LeaferGraphWidgetEntry;

export function createAuthoringModule(options: {
  scope?: { namespace?: string; group?: string };
  nodes: DevNodeClass[];
}): NodeModule;

export function createAuthoringPlugin(options: {
  name: string;
  version?: string;
  scope?: { namespace?: string; group?: string };
  nodes?: DevNodeClass[];
  widgets?: DevWidgetClass[];
}): LeaferGraphNodePlugin;
```

其中：

- `defineAuthoringNode(...)`
  - 把节点类映射成正式 `NodeDefinition`
- `defineAuthoringWidget(...)`
  - 把 widget 类映射成正式 `LeaferGraphWidgetEntry`
- `createAuthoringModule(...)`
  - 给节点类批量加 scope
- `createAuthoringPlugin(...)`
  - 是这个独立包最重要的统一产物
  - 内部负责 widget 注册顺序、module 安装和 plugin 组装

## 8. 进入核心库的方式

### 8.1 直接宿主接入

最直接的使用方式是：

```ts
const graph = createLeaferGraph(container, {
  document,
  plugins: [authoringPlugin]
});
await graph.ready;
```

在这种情况下，这个独立包只是在帮开发者更容易产出 `LeaferGraphNodePlugin`。

如果初始 `document` 里已经出现作者层节点类型，那么 plugin 必须进入启动期 `plugins` 链，或至少先 `graph.use(plugin)` 再 `replaceGraphDocument(document)`；不能等 `graph.ready` 之后再补装，否则启动期文档恢复时还看不到这些类型。

### 8.2 内部示例接入

内部示例、模板工程、演示包也可以直接消费：

- `createAuthoringPlugin(...)`
- `createAuthoringModule(...)`
- `defineAuthoringWidget(...)`

这样内部示例就不必反复手写同样的 `NodeDefinition` / `WidgetEntry` 样板代码。

如果内部示例没有走统一 plugin，而是手工装配 `module + widgets`，顺序必须明确写成：

1. 先注册 widgets
2. 再安装 module
3. 最后恢复依赖这些类型的 `document`

原因很直接：

- 当前节点注册默认会校验节点定义里引用的 widget 类型
- 先装 module、后注册 widget，会在注册阶段直接失败

因此 v1 虽然允许直接消费 `createAuthoringModule(...)` / `defineAuthoringWidget(...)`，但推荐默认仍优先用 `createAuthoringPlugin(...)` 收口顺序。

## 9. 持久化与提交语义

这份方案本身不定义任何外部消费层专属提交协议。

它只约束作者层和核心宿主共有的基本语义：

- `setValue(...)`
  - 本地预览态
- `commitValue(...)`
  - 正式提交边界，由宿主决定如何消费

不同宿主可以有不同策略：

- 直接宿主可以只改运行时
- 某些宿主可以把它转成正式操作

因此这个独立包应做到：

- 不和任何外部消费层私有协议耦死
- 只保持和正式核心宿主语义兼容

## 10. 示例

### 10.1 节点作者示例

```ts
interface EchoProps {
  message: string;
  uppercase: boolean;
  status: string;
}

interface EchoInputs {
  text: string;
}

interface EchoOutputs {
  text: string;
}

export class TextEchoNode extends BaseNode<
  EchoProps,
  EchoInputs,
  EchoOutputs,
  { runCount: number }
> {
  static meta = {
    type: "text-echo",
    title: "Text Echo",
    inputs: [{ name: "text", type: "string" }],
    outputs: [{ name: "text", type: "string" }],
    properties: [
      { name: "message", type: "string", default: "Hello" },
      { name: "uppercase", type: "boolean", default: false },
      { name: "status", type: "string", default: "READY" }
    ],
    widgets: [
      { type: "input", name: "message", value: "Hello" },
      { type: "toggle", name: "uppercase", value: false },
      { type: "button", name: "run", value: "Run", options: { action: "run" } }
    ]
  };

  createState() {
    return { runCount: 0 };
  }

  onExecute(
    ctx: DevNodeContext<
      EchoProps,
      EchoInputs,
      EchoOutputs,
      { runCount: number }
    >
  ) {
    const input = ctx.getInput("text");
    const base =
      typeof input === "string" && input.trim() ? input : ctx.props.message;

    const result = ctx.props.uppercase ? base.toUpperCase() : base;

    ctx.state.runCount += 1;
    ctx.setProp("status", `RUN ${ctx.state.runCount}`);
    ctx.setOutput("text", result);
  }

  onAction(action, _param, _options, ctx) {
    if (action === "run") {
      this.onExecute?.(ctx);
    }
  }
}
```

## 11. 和外部消费层的关系

这套方案不是外部消费层兼容层，而是一层更靠近作者的独立包。

关系建议如下：

- 作者层独立包
  - 负责“怎么写节点、widget 和正式宿主可消费的产物”
- 外部消费层
  - 只负责在包外消费这些正式产物

v1 最稳的接法是：

- 独立包负责产出正式 `LeaferGraphNodePlugin`
- 外部 wrapper、模板工程或其他宿主继续消费这个 plugin
- 不把外部装载链的历史语义回灌进作者层公共 API

## 12. 非目标与边界

- 不把作者层包直接提升为 core 公共模型
- 不修改 `NodeDefinition`、`NodeModule`、`LeaferGraphWidgetEntry` 的正式地位
- 不把 widget 重新塞进 `NodeModule`
- 不承诺一个当前 core 并不存在的节点 destroy 生命周期
- 不让独立包依赖 editor
- 不让独立包承担现有 loader、bundle、quick create 或 bridge 兼容责任
- 不为了作者体验而模糊 `packages/node`、`packages/leafergraph`、`packages/editor` 的边界

## 13. 建议落地顺序

1. 先固化 v1 生命周期支持矩阵与同步合同
2. 再实现 `BaseNode`、`DevNodeContext` 与 `DevNodeClass`
3. 再实现节点类到 `NodeDefinition` 的完整生命周期映射
4. 再实现 `BaseWidget`、`DevWidgetContext` 与 `DevWidgetClass`
5. 再实现统一 `createAuthoringPlugin(...)`
6. 最后再由包外消费层决定是否需要额外 wrapper 或模板接入

## 14. 当前结论

如果后续要补一个真正开发者友好的节点 / widget authoring 包，最稳的方向不是改写 core，也不是把它写成 editor 子系统，而是：

- 保持 core 正式模型不变
- 增加一个独立作者层包
- 让节点和 widget 开发者都走统一 authoring 体验
- 最终输出核心库当前已经支持的正式 plugin / module / widget 注册对象
- 外部消费层若要使用这些产物，只能由外部来兼容它，不能反过来把历史适配负担写回包内

这样既能保住当前架构边界，又能真正简化节点和 widget 开发者的工作流。
